## Problem Statement

当前的 `opsx:auto` 是一条「造零件流水线」——每条内置 pipeline（`full-feature` / `small-feature` / `bug-fix`）都假设任务的产物是**代码变更文档**（propose → apply → verify → ship → archive）。它对「把一个目标反复逼近达标」这类任务无能为力：

- **性能优化**：Lighthouse 刷到 90+、p99 压到 50ms、内存峰值降到阈值——需要「改 → 测 → 再改」闭环，达标点是数字。
- **质量达标**（rubric）：把某模块的错误处理改进到满足给定质量准则——达标点是一份 rubric，由 reviewer 判定。
- **研究式写作**（autoresearch）：研究 → 综合 → 批判 → 精炼，直到质量达标——产物是文档，达标点同样是 rubric。

这三类任务的共同形状是 **harness loop**：重复执行直到停止条件满足。loops.md 博客的核心论点——**确定性的成功判据（score threshold、test count）是最有效的停止条件**——回答了「怎么判达标最好」，但也承认无法量化的目标同样存在。

## What Makes This Cool

关键洞察是一次被推翻的设计：v3 试图把三类任务塞进**一条** pipeline——`measure` 与 `evaluate` 两个闸门可选可组合（AND 语义）、尾部按工作产物 `condition` 分叉、一个 generic iterate skill 同时写代码和散文。对抗式审查发现**三个承重缺陷**（见 §Approaches Considered 的 v3 条目），每一个都能让循环在真实任务上静默失败。

收敛后的 v4 洞察是：**差异是正交的，但不该用「一条 pipeline + 运行时条件」去吸收差异，而该用「一个入口 +一族同构 pipeline」去分流差异**。用户只看到一个命令 `/opsx:goal`；LEAD 据任务分类，选**恰好一条**后端 pipeline；每条 pipeline 只承载一种闸门、一种 iterate 风味、一种尾部——没有条件、没有组合、没有歧义。这复用现有 LEAD 编排、role isolation、run-state/resume、gate 全套基建，而不是另造第二套循环逻辑，也不需要在一条管道里塞运行时分支。

> **学生与考官，无论考数学还是考语文，体系都一样。** 学生始终是 `implementer`，考官在中立命令（measure）或 fresh `reviewer` worker（evaluate）之间二选一——**不新增 role**。差异只在「考官怎么判卷」和「学生交什么」，而这两个轴由管道选择固化，不在运行时再分流。

## Constraints

- **不破坏现有三条 pipeline。** `full-feature` / `small-feature` / `bug-fix` 行为逐字节不变；`review-cycle` loop 零回归（discriminated union 保留其原形状）。
- **数据驱动注册表。** goal-loop 是注册的 pipeline + 一个 loop kind，不硬编码在 `auto.ts`。
- **有界且可中止。** 每个循环都有 `maxRounds` + `loopStallLimit`，绝不无界。
- **author ≠ verifier。** measure = 中立命令判卷；evaluate = fresh reviewer worker ≠ implementer。
- **平层级（flat hierarchy）。** worker（含 goal-iterate implementer）**绝不** spawn 子 subagent。研究由 implementer 内联完成 + 标准 Step H.3 relay，不造 research-sibling。
- **诚实终止。** `maxRounds` 用尽时标注 `outcome: maxRounds-exhausted`，**绝不谎报成功**。
- **产物可审计可恢复。** 每轮判定落盘 `goal-run.json`，是循环位置的权威来源。

## Premises

1. **一个用户入口，一族三条同构后端 pipeline。** `/opsx:goal <task>`；LEAD 分类后选 `goal-loop-measure` / `goal-loop-evaluate` / `goal-loop-research` 之一；显式 override（`measure|evaluate|research` 选择子或 `--pipeline goal-loop-<variant>`）总赢。
2. **每个 pipeline 恰好一种闸门、一种 iterate 风味、一种尾部**（同构 = 无运行时条件）。
3. **`goal` 是 `StageLoopSchema` 的新 discriminated 成员**；闸门本身也是 `measure | evaluate` 的 discriminated union，**必填**——schema 层就排除了组合。
4. **循环体复用现有角色**（implementer / reviewer / planner / shipper，不新增）；每轮 = 单次 implementer dispatch → 正式闸门 → 未达标则同 worker 接着改（warm 复用）。
5. **`goal-run.json` 是循环位置的权威来源**；`auto-run.json` 的 `loopConfig`/`loopProgress` 是派生缓存，冲突时以 goal-run.json 为准。
6. **v1 有界。** `maxRounds`（默认 5）+ `loopStallLimit`（默认 2，gate-neutral，连续无进展轮触发 LEAD 策略评审）。

## Approaches Considered

### Approach v3（已被对抗式审查否决）: 单条 `goal-loop` pipeline + 组合闸门 + 条件尾部

Summary：一个 `goal` loop kind，`measure`/`evaluate` 两个闸门可选可组合（AND），尾部按 `workProduct` 用 `condition` 分叉（code→ship+archive，prose→report），一个 generic iterate skill 同时承载代码与散文。

Effort: M-L | Risk: Med→High（三个承重缺陷）

**三个承重缺陷（每个都能让循环在真实任务上静默失败）：**

- **(a) AND 语义的 stall 漏洞。** 组合闸门（measure AND evaluate）下，「一轮算不算有进展」没有唯一定义：measure 的 score 可能一直达标（平坦），evaluate 一直挂——于是 score 平坦既可能误触发 stall，也可能永不触发（静默烧到 maxRounds）。stall 信号在组合下是坏的。
- **(b) 未强制的条件尾部互斥。** 现有 `condition` 是 freeform label，LEAD 解释，**无 schema 级互斥**——`workProduct=code`/`prose` 两个尾部理论上可能都不跑（交付缺失）或都跑（散文任务被 ship 提交空代码）。
- **(c) 被敷衍的 generic prose skill。** 一个 generic `openspec-goal-iterate` 同时写代码和研究/散文——代码路径清晰，但研究需要 Web search/源收集工具，且「implementer 自评对照 evaluate 目标」与 author≠verifier 相悖。

**v4 如何消解三者：** 一族同构 pipeline，每条恰好一种闸门、一种 iterate 风味、一种尾部——条件、组合、尾部歧义在 schema/管道层就消失，不留到运行时。

### Approach A（v4，已实装）: 单一入口 + LEAD 分流 + 一族三条同构 pipeline

Summary：`/opsx:goal` 一个入口；LEAD 分类选 `goal-loop-measure`（measure 闸门 + code iterate + ship→archive）/ `goal-loop-evaluate`（evaluate 闸门 + code iterate + ship→archive）/ `goal-loop-research`（evaluate 闸门 + prose/research iterate + report 尾部）之一。
Effort: M-L | Risk: Low–Med
Pros: 三个缺陷在管道层消解；现有 pipeline 零改动；measure/evaluate 共享同一套循环机制；分类与 `/opsx:auto` 同构（explicit wins）。
Cons: 三条 pipeline YAML + 三个轻量 skill；discriminated-union 迁移需逐一过消费点。
Reuses: `StageLoopSchema`、`StageRoleSchema`、`ORCHESTRATION_PLAYBOOK`（含 Step L）、run-state、warm-seed resume、gate、ship、archive、stage `condition`（仅给现有 ui/non-ui 用，不给 goal-loop 尾部用）。

### Approach B: 独立 /opsx:goal 命令（脱离 auto 编排）
Effort: S | Risk: Med | Pros: 上手最快 | Cons: 与 auto 的 run-state/resume/worker 体系脱节，系统里出现第二套并行循环逻辑。Reuses: 无。

### Approach C: 全面泛化 auto
Effort: XL | Risk: High | Pros: 愿景最大 | Cons: 动态合成放弃注册表可审计性；scope 蔓延。Reuses: 无。

## Recommended Approach

**Approach A（v4）— 单一 `/opsx:goal` 入口 + LEAD 分流到一族三条同构后端 pipeline。** 已实装。

### 技术设计

#### 1. Schema（`src/core/pipeline-registry/types.ts`）— discriminated union + 闸门必填

`StageLoopSchema` 从 `z.object({ kind: z.literal('review-cycle'), ... })` 改为 discriminated union；`goal` 变体里 `gate` 本身也是 `measure | evaluate` 的 discriminated union 且**必填**——schema 层就排除了组合（你无法在数据里写出 "measure AND evaluate"）。

```ts
export const StageLoopSchema = z.discriminatedUnion('kind', [
  z.object({ kind: z.literal('review-cycle'), maxRounds: z.number().int().positive().default(3) }),
  z.object({
    kind: z.literal('goal'),
    gate: z.discriminatedUnion('kind', [
      z.object({
        kind: z.literal('measure'),
        // Optional in the registry schema, REQUIRED at run-time: the LEAD
        // asserts it is present (read from goal-plan.md) before round 1.
        command: z.string().min(1).optional(),
        threshold: z.number().optional(),                 // score 停止阈值
        target: z.number().optional(),                    // passed-count 目标
        direction: z.enum(['gte', 'lte']).default('gte'), // lte = 越小越好（延迟/内存）
        timeoutSec: z.number().int().positive().default(120),
      }).strict(),
      z.object({
        kind: z.literal('evaluate'),
        // NL success criterion + rubric — injected at run-time from goal-plan.md.
        goal: z.string().min(1).optional(),
        rubric: z.string().optional(),
      }).strict(),
    ]),
    maxRounds: z.number().int().positive().default(5),
    loopStallLimit: z.number().int().positive().default(2), // gate-neutral；避免与 HandoffConfigSchema.stallLimit 撞名
    runArtifact: z.string().default('goal-run.json'),
  }).superRefine((s, ctx) => {
    // The bare registry template `{ kind: measure }` (no command) is ALLOWED:
    // the pipeline registers only the gate type; the LEAD injects the concrete
    // command + threshold/target at run-time. This guard only catches a
    // half-configured measure gate that names a command but lacks a stop condition.
    if (
      s.gate.kind === 'measure' &&
      s.gate.command !== undefined &&
      s.gate.threshold === undefined &&
      s.gate.target === undefined
    )
      ctx.addIssue({ code: 'custom', path: ['gate'], message: 'measure gate with a command needs threshold or target' });
  }),
]);
```

关键点：
- **`gate` 必填且是 discriminated union** → 组合在 schema 层不可能（消解缺陷 a 的根）。
- **字段名 `loopStallLimit`**（不是 `measureStallLimit`，不是 `stallLimit`）。注册表已有 `HandoffConfigSchema.stallLimit`；`loopStallLimit` 是 gate-neutral（对 measure 与 evaluate 的进展都适用），不以某一种闸门命名。
- **measure.command 注册时可选、运行时必填**：注册的 pipeline 只声明闸门*类型*（`{ kind: measure }`）；LEAD 在首轮前从 `goal-plan.md` 注入具体 `command`/`threshold`/`target` 到 `iterate.loopConfig` 并断言 command 存在。管道数据静态，每任务命令可变。

#### 2. 三条同构 pipeline（`pipelines/goal-loop-*/pipeline.yaml`，自动发现）

| pipeline | 闸门（考官） | iterate 风味 | 尾部 | 覆盖 |
|---|---|---|---|---|
| `goal-loop-measure` | measure（确定性命令 → `{score,passed}`） | code-edit | ship → archive | perf、刷分、延迟/内存调优 |
| `goal-loop-evaluate` | evaluate（fresh reviewer → `{satisfied,gaps}`） | code-edit | ship → archive | 对照 rubric 的代码质量 |
| `goal-loop-research` | evaluate（fresh reviewer） | research+prose（web 工具） | report | autoresearch |

- **measure/evaluate** 的尾部是 `ship` → `archive`（各 `model: sonnet`）——迭代的代码正常交付。
- **research** 的尾部是单个 `report` stage（`openspec-goal-report`），**无 ship/archive**——研究产物是文档，无代码可提交。管道还设 `handoff.roles.implementer: 0.35`（更低阈值，更早 relay）。
- **没有 `condition`**：尾部由管道选择固化，不是运行时 label 互斥（消解缺陷 b）。

#### 3. 三个轻量 skill + `/opsx:goal` 入口

- **`openspec-goal-plan`**（planner，`define-goal` stage）：输入任务描述 → 产出 `goal-plan.md`（`goal` / `gate` / `workProduct` / `maxRounds`）。`gate: true` 让用户人工确认 measure 命令（顺带是「measure.command 是任意 shell」的安全阀）。不产出 proposal/design/specs。
- **`openspec-goal-iterate`**（implementer，`iterate` loop stage）：工作产物感知的 dispatch 指令——code 改代码、prose 做研究+写作。**绝不 spawn 子 subagent**（平层级）；研究内联完成，context 满时走标准 Step H.3 self-handoff（写 handoff doc，返回 HANDOFF），LEAD warm-seed 接力者，循环继续——`goal-run.json` 是跨 relay 存活的脊柱。**这是「implementer 内联 + relay」决策**，不是 research-sibling（消解缺陷 c）。
- **`openspec-goal-report`**（shipper，`report` stage，仅 research）：把 `goal-run.json` 汇总成最终文档。
- **`/opsx:goal` 入口**（`openspec-opsx-goal` skill + `OPSX: Goal` command）：跑 LEAD pre-flight + 分类，选一条后端 pipeline，然后驱动**同一套** `ORCHESTRATION_PLAYBOOK`（内嵌，不重新实现编排）。分类关键词：`score|latency|optimize|lighthouse|benchmark|p99|memory|throughput`→measure；`rubric|quality|clean|standard|refactor-quality`→evaluate；`research|investigate|write report|write brief|autoresearch|literature`→research；歧义默认 evaluate。

#### 4. LEAD playbook Step L — Goal-loop（`_orchestration.ts` 新增；Step E 的 preamble 改为按 `loop.kind` 分流）

单次 dispatch 每轮模型（与 review-cycle 同构）：

- **注入**（首轮前一次）：读 `goal-plan.md`，把具体闸门配置合并进 `iterate.loopConfig`；断言 measure 闸门有 `command`。
- **每轮**：
  - dispatch `implementer` worker（**warm 复用所有轮**——同一 worker，与 review-cycle 复用 fixer thread 同款；轮次不各自消耗新 relay）。种子：第 1 轮 = goal-plan.md（无历史分）；第 N>1 轮 = goal-plan + 前轮结构化反馈 `{score/gaps, measurePassed/evaluateSatisfied}`。implementer 可在 dispatch 内自由自测；本轮**正式分数**是返回后的闸门判定。
  - **闸门**（按管道的单一闸门类型）：
    - measure：运行 `gate.command`，解析 `{score, passed, detail}`。**失败分支**：非零退出/超时/坏 JSON → 记 `{round, error}`，该轮 not-passed，把 stderr 当 gap 喂回。**不 deadlock**。
    - evaluate：dispatch **fresh reviewer worker**（≠ implementer，author≠verifier），返回结构化 `{satisfied: boolean, gaps: string[]}`（不接受自由文本）。
  - 记录 `{round, score?, measurePassed?, evaluateSatisfied?, detail?, gaps?, error?, gitTreeFingerprint}` 追加进 `goal-run.json`。
- **停止**：闸门 satisfied → 进尾部。`maxRounds` 用尽 → 进尾部但标注 `outcome: maxRounds-exhausted`，**绝不谎报成功**。
- **stall**（gate-neutral，per-axis）：一轮算有进展 = (measure 在且 score 朝目标移动——gte 升 / lte 降) 或 (evaluate 在且 gap 收缩/新达标)。第 1 轮总算进展。`loopStallLimit` 连续无进展轮 → LEAD 策略评审（warm-seed 换思路，或 escalate）。**不静默烧轮。**
- **resume**（权威来源 = `goal-run.json` 最后一条记录）：
  - 最后一条 satisfied → 直接进尾部，**不重跑**。
  - 最后一条未达标（该轮已判、有记录）→ 该轮完成；从 **`lastRound + 1`** 起跑（fresh dispatch，种子带前轮 gap）。不是重跑 N——N 已有记录的判定。
  - **无记录**（define-goal 完成、iterate 起步即死，首判未跑）→ dispatch 第 1 轮。
  - 进轮前可先重判一次（抓 flaky 命令 / 产物被外部改达标）；`gitTreeFingerprint` 让 resume 检测 tree 是否在自己之下被改动。

#### 5. run-state 扩展（纯加法，`passthrough()`）

```ts
// 注入的实际 loop 配置，运行时权威
loopConfig?: {
  kind: 'goal';
  gate: { kind: 'measure'; command: string; threshold?: number; target?: number; direction: 'gte'|'lte' }
       | { kind: 'evaluate'; goal: string; rubric?: string };
  maxRounds: number;
  loopStallLimit: number;
  workProduct: 'code' | 'prose';
};
// best-effort 派生缓存；权威位置在 goal-run.json
loopProgress?: {
  kind: 'goal';
  round: number;
  lastScore?: number;
  measurePassed?: boolean;      // measure 闸门在时出现
  evaluateSatisfied?: boolean;  // evaluate 闸门在时出现
  stallStreak: number;
  historyRef: string;           // -> goal-run.json
};
```

per-axis stall → 两个独立布尔（每种闸门只出现自己的），不存在 v3 组合下的 AND 语义歧义。

## Open Questions（v1 决议）

1. **measure.command 沙箱** → v1 靠 `define-goal` 的 gate 人工确认测量命令拦危险命令；v2 再加强制沙箱。
2. **implementer 自测工具化** → v1 让 implementer 在 dispatch 内直接 Bash 跑 measure 命令；v2 若可观测性不足再加命名工具 `opsx goal measure`。
3. **每管道 maxRounds 默认** → research/evaluate 可在管道 YAML 的 `loop.maxRounds` 设更低默认（如 3）；schema 默认仍是 5。
4. **goal-loop 与 decompose 组合** → v1 不禁止 portfolio 子 change 各自是 goal-loop，但不显式测试。

## Success Criteria（已达成）

- `openspec pipeline list --json` 列出三条 goal-loop pipeline；`openspec pipeline show goal-loop-<variant> --json` 返回有效 DAG。
- measure 任务跑通 define-goal → iterate（多轮，goal-run.json 记录每轮）→ ship → archive。
- measure 命令失败（非零/坏 JSON）不 deadlock：记 error、按 not-passed 继续。
- evaluate 任务跑通，fresh reviewer 每轮判 `{satisfied, gaps}`。
- research 任务跑通到 `report` 尾部（无 ship/archive）。
- `maxRounds` 用尽时尾部标注 `outcome: maxRounds-exhausted`，不谎报成功。
- `loopStallLimit` 连续无进展轮触发 LEAD 策略评审。
- 中途 kill 后 `openspec pipeline resume <change>` 按精确规则恢复（satisfied→尾部；未达标→lastRound+1；无记录→第 1 轮）。
- 现有 `small-feature` / `bug-fix` / `full-feature` / `review-cycle` 测试零回归。

- 你没有说「我要做个 autoresearch 工具」——你说的是「现有的 auto 已经很好，但它太专」。你是在已经跑通的、令你满意的系统上找它的边界。
- 你一口气甩了三个参考（loops.md 博客、autoresearch、/goal），不是要照搬某一个，而是用它们 triangulate 一个形状。
- 你把想扩展的东西指给一个抽象层次（「pipeline 注册表原语」），说明你认同数据驱动哲学——要的是和现有体系同构的东西。
- **而 v3 → v4 的收敛，是被对抗式审查逼出来的**：v3 的「一条管道吸收所有差异」听起来最优雅，但三个承重缺陷每一个都能让循环在真实任务上静默失败——AND 语义的 stall 漏洞、未强制的条件尾部、被敷衍的 generic skill。你没有用「加更多运行时检查」去打补丁，而是把差异**从运行时条件移到管道选择层**——一族同构管道，每条自洽。这是「让一台机器真正通用」的正确姿势：不是一条管道塞满开关，而是用分类把任务送到已经为它量身定制的那条管道。
