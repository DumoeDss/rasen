## 背景与动机(Problem Statement)

rasen 目前为每个 workflow 维护**两份几乎相同的 prompt body**:skill 版(`.claude/skills/rasen-*/SKILL.md`)和 command 版(`.claude/commands/rasen/*.md`),分别出自同一模板文件里的两个模板函数(如 `src/core/templates/workflows/apply-change.ts` 中的 `getApplyChangeSkillTemplate` 与 `getOpsxApplyCommandTemplate`)。这带来三个问题:

1. **双份维护与漂移**。两份 body 手工同步,已出现轻微内容漂移(如 apply 的 contextFiles 说明两版括号举例不一致)。
2. **官方架构已合并**。Claude Code 官方文档(2026-07 抓取确认)明确:"**Custom commands have been merged into skills.** A file at `.claude/commands/deploy.md` and a skill at `.claude/skills/deploy/SKILL.md` both create `/deploy` and work the same way. Your existing `.claude/commands/` files keep working. … **Skills are recommended** since they support additional features like supporting files." skill 目录名即 slash 命令名(`.claude/skills/rasen-apply-change/` → `/rasen-apply-change`),command 目录只是仍兼容的旧布局。
3. **写死的上下游 steering 在子集安装下悬空**。skill body 里硬编码了链条引导(如 apply 的 "state: all_done → steer to `/rasen:verify` → `/rasen:ship`"),而 core profile 只装 `propose/explore/apply/sync/archive/auto/help`——**不含 verify 和 ship**,提示指向不存在的 skill。这也违反 `src/core/templates/workflows/_orchestration.ts` 头注释里自己声明的 "HOW(playbook)与 WHAT(pipeline DAG)解耦" 原则:LEAD 编排的 DAG 已在 pipeline registry,skill 里再嵌一份链条是第二事实源。

## 决策(全部【已拍板】)

1. **移除 commands 交付面,全工具一刀切**。delivery 收敛为 skills-only;`src/core/command-generation/` 全模块(26 个 tool adapter)删除;各 workflow 模板的 CommandTemplate 函数删除。不保留死架构、不做 per-tool 保留。真有需求时从 git 历史恢复。
2. **上下游衔接改由 CLI 运行时提供,数据源为新建的独立链条表**。不挂 pipeline registry(那是自动流水线 DAG,gate/loop/角色字段对交互场景是噪声),不扩展 schema 状态机(schema 管 artifact 产物层,塞入工作流动作是概念扩张)。新建一张轻量"workflow 衔接表",`rasen instructions`/`rasen status` 运行时读表 + 已装 workflow 集合过滤后输出 `nextSteps`。
3. **skill body 交叉引用统一为 canonical skill 名**(`rasen-continue-change` 而非 `/rasen:continue`),配一句"用本工具的调用名转述"。
4. **dispatched 契约(expert `_shared.ts` 的 PREAMBLE、"Dispatched vs standalone mode")留在 skill 不动**——它是 skill 作为可编排单元的接口契约(HOW),不是链条知识(WHAT)。明确 non-goal。

**已接受的代价**:Claude Code 上丢失 `/rasen:apply` 冒号短名,调用名变为 `/rasen-apply-change`(skill 目录名)。项目级 skill 拿不到冒号命名空间(只有 plugin 打包或 commands 子目录能产生),用户已确认接受。

## 约束(Constraints)

- **版本号不动**:不 bump major/minor,发布类改动版本无关(读 `package.json`)。
- **legacy config 兼容**:全局配置里既有 `delivery` 值(`both`/`commands`/`commands-first`/`skills-first`)读到时**绝不报错**,静默映射 + 一次性 notice + 回写,沿用 `src/core/global-config.ts:55-68` 的 `normalizeDelivery`/`LEGACY_DELIVERY_MAP` 既有模式(profiles spec 已有 retired-value 先例)。
- **存量项目清理**:`rasen update` 必须删除既存的 command 文件,复用 `src/core/update.ts:561` 附近既有的 removeCommandFiles 机制(它今天在 delivery=skills 时已做同样的事,改为无条件执行)。
- **i18n 三语**:所有新增/改动的用户可见文案同步 `src/locales/en.json`、`ja.json`、`zh-cn.json`。
- **spec 同步守卫**:改 `rasen/specs/` 时,场景删除/改名必须用 REMOVED + ADDED 双条目,validate 不查同步场景守卫但 archive 会拒(既有教训)。

## Premises(声明的假设与已验证事实)

- **【已验证,2026-07 专项调研】主流工具已原生支持 Agent Skills 标准**(agentskills.io):Cursor、Windsurf、OpenAI Codex CLI、Gemini CLI、GitHub Copilot、Cline、RooCode、Kilo Code、OpenCode、Qwen Code、Kiro、Goose、Auggie、Crush、Factory Droid 等 15+ 工具均有官方文档确认自动发现项目 skills 目录并按 description 触发,多数还内建手动调用(Cursor/Crush 的 `disable-model-invocation`、Auggie 自动把 skill 注册为同名 slash command、Codex 的 `$skill-name`)。**删除 command 文件后这些工具的调用不受影响,且无需在 AGENTS.md/.cursorrules 等 instructions 文件里注入 skills 索引**——对原生支持的工具注入是多余的(重复占用 context),实现者不要加这个"补救"。
- **已知局限(接受,不补救)**:① Windsurf 的 `.windsurf/workflows/*.md` 是独立于 skills 的纯手动 slash 通道,删除后 Windsurf 用户失去"不依赖模型判断的精确重放"入口(skills 仍可 `@mention`);② Gemini CLI 的 `.gemini/commands/*.toml` 是参数化 prompt 模板系统,skills 不对等替代;③ iFlow、Continue、Amazon Q 的支持细节未确认(证据薄弱),标注为已知空白。以上均不阻塞一刀切决策。
- pipeline registry 与 LEAD 编排 playbook 本次完全不动。
- user-authored workflow(workflow-registry 中 `source === 'user'`)的既有包若含 command 内容,安装时忽略即可,不需要迁移工具。

## 改动清单(Work Breakdown)

三个 Phase。**A 与 B 可并行;C 依赖 A(引用改名)和 B(转述 nextSteps 的槽位),必须最后做。** 建议每个 Phase 一个 rasen change,或 A∥B 后合并做 C。

### Phase A — 移除 commands 交付面

- **A1. delivery 配置整个退役**(`src/core/global-config.ts`,单一路径,不做"收敛到单值"的中间态)
  - `Delivery` 类型、`DEFAULT_CONFIG.delivery`(:158)、`LEGACY_DELIVERY_MAP` 全部删除。
  - `normalizeDelivery` 由"归一化"改为"退役检测":读取端遇到配置里存在 `delivery` key(**无论值是什么**,包括当前 live 值 `'both'`/`'skills'` 与旧 legacy 值 `commands`/`commands-first`/`skills-first`)→ 一次性 notice("commands have been consolidated into skills; the delivery setting has been retired")+ 从配置对象删除该 key + 回写。注意现在 `normalizeDelivery`(:61-68)把 `'both'` 当作当前值直接返回——这个分支随类型一起删,不能只在 map 里加条目。
  - notice 文案进三语 locale,风格对齐既有 "has been consolidated into" 文案。
- **A2. init 接线**(`src/core/init.ts`)
  - 删 `:705-726` 的 delivery 读取与 `shouldGenerateCommands`;删 `:783-815` 的 `generateCommands` 调用分支;删 `:902-944` 的 commandCount 统计与 "Removed: N command files" 输出;`syncWorkflowArtifactLedger` 的 delivery 参数按 B 侧 ledger 现状同步收敛。
  - init 在全新项目**只生成 skills**,并顺手清理目标目录里可能存在的旧 command 文件(同 A3 机制)。
- **A3. update 接线与存量清理**(`src/core/update.ts`)
  - 删 `:117-168`、`:263-337` 的 delivery 分支与 `generateCommands` 调用。
  - **清理涉及两个不同的方法,勿混淆**:`removeCommandFiles`(:564-590,原 delivery=skills 时删已选 workflow 的 command 文件)与 `removeUnselectedCommandFiles`(:597-637,含 stalePaths 与 legacy `-command` 后缀清理)。两者合并/收敛为一个**无条件执行**的存量清理步骤:对每个检测到的工具,删除全部内置 workflow 的 command 文件 + legacy 后缀变体。init.ts 也有同名清理方法(:1017 附近),同步处理。保留"绝不删除用户自有文件、只删 known-id 文件"的既有安全边界。
  - **清理的路径解析不得依赖将被删除的 live 数据**:今天这些方法依赖 `command-generation/` 的 `CommandAdapterRegistry`/`getCommandFileId`/`getCommandFilePathCandidates`/`getLegacyCommandFilePath`,以及 workflow definition 的 `command` 字段(A6/A7 会删)——若不处理,清理会静默清零。修法见 A4。
  - 更新输出文案:"Removed: N command files (commands have been consolidated into skills)"。
- **A4. 删除 command-generation 模块**(`src/core/command-generation/` 整目录),**先抽出清理所需的静态知识**
  - 仿照既有先例 `legacy-cleanup.ts:70` 的 `RETIRED_WORKFLOW_COMMAND_IDS`(静态 ID 列表 + 路径候选解析):把**全部内置 workflow 的 command ID 冻结为静态清单**(19 个),连同各工具的 command 目录/文件名规则(即 `getCommandFileId`/`getCommandFilePathCandidates`/`getLegacyCommandFilePath` 与 adapter 里的路径知识)迁入一个保留的清理专用模块(如 `src/core/shared/retired-command-paths.ts`)。A3 的清理只读这份静态数据,不再依赖 live registry 或 `definition.command`。
  - 迁移 `yaml.ts` 的 `quoteYamlValue`(被 `skill-generation.ts:12` 引用)到 shared(如 `src/core/shared/yaml.ts`);`skill-generation.ts:11` 的 `CommandContent` import 随 A5 一并删除。
  - 然后删除整目录:26 个 adapter + `generator.ts`/`registry.ts`/`types.ts`/`toml.ts`/`command-file-id.ts`/`yaml.ts`。全仓 grep `command-generation` 确认零残留 import——已知另有两个消费者会被牵出:`migration.ts:48` 与 `profile-sync-drift.ts`(:8-9/:60/:185/:196/:253,并行的 command 漂移清理路径,也用 `definition.command.content.id`),两者的 command 清理逻辑同样改读 A4 的静态清单(类型删除会在编译期强制暴露,不会静默漏掉)。
- **A5. skill-generation 清理**(`src/core/shared/skill-generation.ts`)
  - 删 `CommandTemplateEntry`(:33-42)、`getCommandTemplates`(:153-175)、`getCommandContents`(:182-192)及 `CommandContent` 类型引用。
- **A6. 模板层清理**(`src/core/templates/`)
  - 每个 `workflows/*.ts` 删除 `get*CommandTemplate` 导出(共 19 个);`types.ts` 删 `CommandTemplate` 类型;`index.ts` 清导出。**skill 模板正文本 Phase 不动**(正文改动全部归 Phase C,避免两个 Phase 改同一批文件冲突)。
- **A7. workflow-registry 与 workflow 包**
  - workflow definition 的可选 `command` 字段退役:catalog 加载端忽略该字段(带一条 debug 级提示),类型层删除。
  - `workflow-author` / `workflow-review` 两个专家模板中涉及"生成/审查 command"的段落同步删改(此两处属专家自身职责描述,归 A 不归 C)。
- **A8. 交互面与文案**
  - `rasen profile` / `rasen config` 中 delivery 相关 prompt、结果展示、帮助文本删除(`src/commands/config*.ts`、profile 命令组);三语 locale 同步。
- **A9. specs 更新**(delta spec,注意 REMOVED+ADDED 守卫)
  - `rasen/specs/profiles/spec.md`:delivery Requirement 改写为 skills-only + legacy 映射(retired 值清单加入 `both`)。
  - `rasen/specs/cli-init/spec.md`、`cli-update/spec.md`:command 生成场景 REMOVED,清理场景 ADDED。
  - 全 specs grep `command`(区分 CLI 子命令语义与交付面语义,只改后者)。
- **A10. 测试**
  - 删 command-generation 相关测试;init/update 测试改断言(零 command 文件、清理行为、legacy 映射 notice);新增"update 清理存量 command 文件且不动用户自有文件"用例。
- **A11. 明确 non-goal:不做 instructions 注入**
  - 不在任何工具的 instructions 文件(AGENTS.md、.cursorrules、GEMINI.md 等)注入 skills 索引——调研确认原生支持工具会自动扫描,注入属重复;未确认的工具(iFlow/Continue)按已知空白处理。
- **A12. 文档**
  - `docs/commands.md`、`docs/how-commands-work.md`、`docs/supported-tools.md`、`docs/customization.md`、`docs/getting-started.md` 等全文更新;`docs/zh/` 若有对应文件同步。CHANGELOG 条目(版本归属由用户定)。

### Phase B — CLI 运行时 nextSteps

- **B1. 新建链条表**(建议 `src/core/workflow-chain.ts`,命名【实现者裁量】)
  - 静态表:`canonical workflow id → { next: id[], 触发条件 }`。覆盖交互主链 `propose → (new/continue) → apply → verify → ship → archive`,旁支 `explore`、`office-hours → propose`、`sync`。条件示例:apply 的 next 在 `state=all_done` 时为 `verify`,`blocked` 时为 `continue`。
  - 表是纯数据 + 一个解析函数 `resolveNextSteps(workflowId, state, installedWorkflows)`,返回按已装集合过滤后的 canonical id 列表;若直接后继未安装,**沿链前跳到最近的已装节点**(core profile 无 verify/ship 时 apply 完直接提示 archive)。
- **B2. instructions/status 输出**(`src/commands/workflow/instructions.ts` 及 status 实现)
  - `--json` 增加 `nextSteps: [{ workflow: "<canonical id>", reason: "<state 触发原因>" }]` 字段。
  - **已装集合来源:profile/config 解析,即 `update.ts:154-160` 使用的 `resolveDesiredWorkflowSelection(catalog, profile, config.workflows, ...)`(或等价的 `resolveWorkflowSelection`)。不得用 workflow artifact ledger 作为主源**——ledger 只记录 `source === 'user'` 的自研 workflow(`workflow-artifact-ledger.ts:288` 明确 filter),内置链条 workflow(propose/apply/verify/ship/archive)永远不在其中,用它过滤会把全部内置 nextSteps 判为未安装。
- **B3. stdout hint 双通道**
  - 人类可读输出尾部附 `Next: <workflow id> — <一句话>`,风格对齐既有 hint 惯例("Hints printed by commands already carry the right flag"——store/project 场景下 hint 需带同样的 `--store`/`--project` flag)。
- **B4. i18n**:nextSteps reason 与 hint 文案三语。
- **B5. 测试**:全量安装、core 子集安装(无 verify/ship)、单 workflow 安装三种矩阵下的 nextSteps 断言;blocked/all_done 状态分支。

### Phase C — skill 模板正文清洗(依赖 A、B)

- **C1. 剔除写死 steering**
  - 全模板 grep `/rasen:` 与 `steer to`,删除所有硬编码链条引导(apply/verify/ship/archive/propose/continue/new 等模板的 "Next:" 段落与 all_done/blocked 分支里的具体指向)。
  - 替换为统一槽位文案(各模板措辞可微调):"展示 CLI 输出中的 nextSteps(用本工具对该 skill 的调用名转述);若本轮尚未调用过带 nextSteps 的命令,先运行 `rasen status --change \"<name>\" --json` 获取。"——后半句是零 CLI 路径的保底。
- **C2. 交叉引用统一 canonical skill 名**
  - `/rasen:apply <other>` 之类示例 → `/rasen-apply-change <other>`(Claude Code 上 skill 即 slash 命令);`consult /tdd` → `consult the rasen-tdd skill`;expert 引用同理。
- **C3. non-goal 确认**:expert `_shared.ts` PREAMBLE、"Dispatched vs standalone mode"、report 落盘路径等 dispatched 契约**零改动**。
- **C4. 测试**:模板快照/断言测试更新;新增 grep 型断言——生成产物中不得出现 `/rasen:` 冒号引用(白名单:历史文档)。

## 验收标准(整体)

1. `rasen init` 全新项目:只生成 skills,`.claude/commands/`(及各工具等价目录)零 rasen 文件。
2. `rasen update` 存量项目:既有 rasen command 文件被清理并输出 notice;用户自有非 rasen 文件不受影响。
3. 全局配置含任意 legacy delivery 值:命令不失败,映射 + 一次性 notice + 回写。
4. `rasen instructions apply --json` 含 `nextSteps`;core profile(无 verify/ship)下 apply 完成时 nextSteps 指向链上实际已装的下一站,不指向未安装 workflow。
5. 生成的 SKILL.md 中无任何写死的 `/rasen:` 冒号引用与硬编码链条指向。
6. `pnpm test` 全绿(Windows 上 CLI-spawning 测试的既知 EBUSY flake 按既有惯例隔离重跑判定)。
