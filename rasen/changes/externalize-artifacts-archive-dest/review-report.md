## Findings

### Major

**M1. Porcelain clean-check is blind to gitignored content — prune can delete the only copy.**
`isChangeDirClean` (src/core/archive.ts:190-197) and the identical template prose (`git status --porcelain -- <changeRoot>`, archive-change.ts step 5, bulk-archive-change.ts step 8b) treat an empty porcelain output as "content is in git history". Ignored files are invisible to porcelain (no `--ignored`), so a change directory covered by `.gitignore` (e.g. a user who ignores `rasen/` — upstream users debate committing the workspace) reads CLEAN, and `prune` then `fs.rm -rf`s the only copy on disk with git history holding nothing. This defeats the exact guarantee D5 states ("uncommitted change-dir content is NOT in git history yet, and destroying the only copy is never acceptable"). Untracked-but-unignored content IS caught (`??` rows; verified by the never-committed fixture in test/core/archive.test.ts "blocks external destination when the change directory has uncommitted content"). Fix direction: additionally require the directory to be tracked (`git ls-files -- <changeDir>` non-empty) or run the check with `--ignored`, in both CLI and template prose. Not Blocker only because the path needs an opt-in prune config plus a gitignored workspace; the failure is silent data loss when it hits.

**M2. No prune tombstone is ever written on the on-merge/CLI path — the default-timing prune is unrecognizable afterwards; plus a marker-token mismatch.**
The tombstone read exists in three places (archive-change.ts step 1.5b ×2 getters; guardrails), but the only WRITER is ship.ts's in-ship branch (`**Pruned in ship:** true`). The archive skill's own step-5 prune branch (`rm -rf` + commit guidance) and the CLI prune branch (src/core/archive.ts:614-645) record nothing in the workDir ship-log. Under the DEFAULT timing (on-merge), a pruned change therefore leaves no record: re-invoking archive falls through 1.5a (no in-repo dir), 1.5b (no external dir, no tombstone line), proceeds to step 2, and status hard-throws "not found" — the loud-triage path — instead of the spec's "report the pruned state and stop cleanly" (archive-destination delta, "Already-archived detection covers all destinations"; opsx-archive-skill delta, "Pruned change is recognized by its record"). Design D3 explicitly names "the prune tombstone (ship-log `Archived:`/`Pruned:` record)" as the third detection leg — that leg only exists for in-ship prunes. Compounding: step 1.5b greps for a `Pruned:` line, but ship.ts writes `**Pruned in ship:** true` — the literal token `Pruned:` does not occur in that line (a literal grep misses it; only LLM leniency saves it). Fix direction: step 5's prune branch (skill + bulk) and the CLI prune branch should append a `Pruned:` line to the workDir ship-log when a workDir exists; unify the marker token between writer and scanner.

**M3. `--yes` conflation: the timing guard's own refusal message instructs the user to pass the flag that silently waives the prune deletion prompt.**
One flag gates two distinct consents: src/core/archive.ts:310-325 (timing guard, skipped entirely under `options.yes`) and :615-637 (prune confirmation — the interactive red prompt is inside `if (!options.yes)`, so `--yes` skips it in BOTH json and interactive modes). Concrete scenario: destination=prune, timing=on-merge, pr-delivered ship-log. First run refuses with fix text "rerun with --yes after confirming the merge yourself". The user complies — and the prune-naming confirmation the spec demands never appears; deletion is immediate. The user consented to "merge is confirmed", not "permanently delete". The specs do name `--yes` for both gates individually, so this is a design-level footgun rather than a spec violation — but the guard's fix message actively steers users onto it. Blast radius is bounded by M1's check (content committed → git-recoverable), EXCEPT when combined with M1's gitignore hole, where the pair is unrecoverable. Fix direction: when destination=prune, the timing-guard fix message should warn that `--yes` will also authorize the deletion; or use a distinct prune override.

**M4. Unimplemented spec SHALL: the machine-home archive GC lifecycle is stated nowhere in doctor/GC documentation.**
archive-destination delta, requirement "Machine-home archives share the home's lifecycle": "This lifecycle SHALL be stated in the doctor/GC documentation for the machine home." Nothing in the diff touches doctor or docs; `doctor --gc`'s help text (src/commands/doctor.ts:275, "Remove dangling machine-home registry entries and their orphaned home directories") says nothing about external archives dying with an unregistered home. Grep across src/ and docs/ finds no such statement. The behavioral scenario (GC spares live homes) holds by child-1 construction, but once this delta syncs to main specs, the spec will assert documentation that does not exist. Strictly this maps to "required spec behavior missing" (Blocker vocabulary); rated Major because it is a one-line documentation fix (extend the `--gc` help/doctor output or docs) — or trim the sentence from the delta before sync. LEAD should treat it as must-resolve-before-archive either way.

### Minor

**m5. Refusal after side effects: the destructive preconditions and prune confirmation run AFTER spec sync.** In run(), specs are synced (~line 500s) before the dirty check (:596) and prune confirm (:615). A refused/declined prune prints "Archive cancelled." with main specs already modified in the working tree, unmentioned. Pre-existing shape (the archive-exists collision error also fires post-sync), but the new gates widen the refusal surface; the cancel/blocked output should note the sync already happened.

**m6. `ensure:true` registry mint fires before every gate.** Destination resolution (src/core/archive.ts:301) runs at the very top of run(): an `external` archive that is subsequently refused (timing guard, validation, dirty tree) has already minted identity, written the registry, and created a home directory — a write on a failure path, against the "ensure only at write time" intent. Benign (mint-once), but moving resolution after the gates (or deferring ensure to the move) would honor the design literally.

**m7. Post-bookkeeping commit guidance misses untracked new spec files; bulk has no commit guidance at all.** `git commit -- "<changeRoot>" "<specsDir>"` (archive-change.ts step 5, both getters) commits tracked deletions/modifications only — a spec sync that CREATES a new capability directory (this very change would!) leaves those files untracked and uncommitted, so the tree does NOT "end clean" (D5's stated outcome) and the archive commit silently omits part of the sync. Needs a preceding `git add <changeRoot> <specsDir>`. bulk-archive-change.ts got preconditions + branches but no commit guidance despite the archive-destination scenario "Removal is committed without archive additions" covering every bookkeeping actor (D7).

**m8. Step 2.5's inconsistency HARD STOP was not extended to the prune marker.** Its branches key exclusively on `Archived in ship:` (archive-change.ts:67-71). A recycled-name change whose stale workDir ship-log carries `Pruned in ship:` + `Mode: pr` sails through the "no Archived in ship: line, Mode: pr" branch — the defense-in-depth stop the recycled-name note promises never fires for the new marker variant. Misleading, not destructive (the active directory routes to step 2 regardless).

**m9. Spec-text tension on the git boundary.** cli-archive delta: "Because the CLI never invokes `gh` or git for workflow decisions…" — yet the new dirty gate shells `execFile('git', ['status', '--porcelain', …])` (src/core/archive.ts:192) and refuses on its result, which is a workflow decision by any plain reading. design.md D5 explicitly prescribes this exact check and tasks 3.3 carves it out as "a local read", so the implementation follows the design of record — but the sentence as it will sync into main specs is false as written. Reword the spec rationale (e.g. "never invokes gh, and uses git only for a local read-only status check") before sync.

**m10. Coverage gaps (all report-only per dispatched mode):** (a) the CLI external→in-repo fallback branch (`destinationFallback`, archive.ts:652-658) has no test — simulatable with a corrupt registry + ensure failure; (b) `isChangeDirClean` null branch (no git repo → "could not verify" refusal) untested; (c) the timing guard's workDir-FIRST ship-log precedence is untested (only the changeDir fallback is exercised — a candidate-order regression would pass the current suite); (d) interactive prune decline (confirm → false → "Archive cancelled") untested.
