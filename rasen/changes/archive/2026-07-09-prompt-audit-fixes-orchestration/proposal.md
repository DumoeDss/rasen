## Why

The orchestration playbook (`_orchestration.ts`) is the LEAD's operating manual, and its taxonomy and accounting have fifteen conflicts (OR-1..OR-15) plus three shared-block seams (SH-4, SH-5, SH-7). The worst fire on the common autonomous path: (OR-1/SH-4, Critical) the planner-reuse decision is governed by two rules that give opposite answers — H.2 compares occupancy to the handoff threshold (0.5), B.1.5 to the reuse threshold (0.25) — so the LEAD over-retains a bloated planner across every portfolio child. The worker-death handling collapses distinct situations into one expensive branch: an infra/transient death (API error, timeout) is treated identically to a context death, cold-reconstructing a fresh worker and burning relay budget when a single `SendMessage` would revive the still-intact agent (the pre-designed H.4 fix); and a `DONE` with a few unticked tasks is likewise mis-classified as a death (OR-6). Counters that both default to the same value are never disambiguated (review rounds vs relays vs strategy attempts vs goal rounds; worker-relay soft-review-at-N+1 vs session-relay hard-stop-at-N — OR-9/OR-13/OR-15). Goal-loop config fields the pipelines set are silently ignored (planner's `maxRounds`, `runArtifact` — OR-5/OR-10), Tier-C has no defined degradation for the `evaluate` gate or `parallelGroup` (OR-4/OR-14), portfolio child-pipeline gates are undefined against "proceed automatically" (OR-3), and several yaml/schema fields are dead or self-contradictory (`verifyPolicy` standard/light, reuse-threshold "headroom" vs occupancy — OR-11/OR-8). Finally the PREAMBLE's solo "Default to action" bleeds into dispatched leaf workers (SH-5) and the LEAD's "do NOT author" opener contradicts its own trivial-inline-fix step (SH-7).

## What Changes

- **Planner/reuse threshold disambiguation (OR-1, SH-4).** State the general rule once: a **cross-change re-staffing** decision (planner reuse, cross-child implementer reuse) compares occupancy to the **reuse** threshold (default 0.25); a **mid-task relay** decision compares to the **handoff** threshold (default 0.5). H.2 inline-exempts planner/cross-child reuse and forward-references B.1.5 / G.1.3.
- **Worker-death taxonomy + infra-death revival (H.4, OR-6).** Add three death classes: (a) **context death** (HANDOFF / compaction) → document relay, unchanged; (b) **infra/transient death** (API error, timeout, died without returning, transcript intact) → FIRST action is `SendMessage` to wake the SAME agent ("interrupted by infra failure; disk may have moved — re-read tasks.md / git status, then continue"), with backoff during overload waves; infra revivals consume NEITHER `maxRelays` NOR `stallLimit`; (c) **transcript lost** → cold reconstruct (only here), record the degradation. `DONE` with unticked tasks is NOT a death — probe/wake the same worker to finish or explain before any reconstruction. Scope F.1's "dead handles" to cross-session explicitly and promote the buried "SendMessage IS transcript-resume" note into the rule body.
- **Counter disambiguation (OR-9, OR-13, OR-15).** One table naming every counter — relay count, review rounds, strategy attempts, goal-loop rounds — their independence, and the maxRelays asymmetry (worker relay = soft review after `maxRelays`; session relay = hard stop at `maxRelays`). Step E gains the goal-loop's "a round may span multiple relays; caps are independent" clarification.
- **Run-state `sessionHandoff.n` (OR-2).** Add `"n": 1` to the canonical run-state example so the session-relay generation cap (H.7) can actually trip.
- **Loop-stage per-role threshold resolution (OR-7).** Inside a loop stage, the LEAD resolves each dispatched worker's handoff threshold by that worker's ACTUAL role, not the loop stage's nominal `role`.
- **Portfolio child-gate semantics (OR-3).** Step G states explicitly how childPipeline internal gates resolve under portfolio orchestration (per the parent's directive), reconciling "proceeds automatically" (which governs the decompose decision only) with "always pause at gate stages."
- **Tier-C degradations (OR-4, OR-14).** Define the evaluate-gate Tier-C author≠verifier fallback (a second freshly-reset single-context pass seeded only with goal+rubric+artifact) and the parallelGroup Tier-C fallback (run members sequentially in-context, collect all results).
- **Goal-loop config holes (OR-5, OR-10, OR-12).** Step L Inject copies `maxRounds` (and `loopStallLimit`) from goal-plan.md; Step L reads `loop.runArtifact` (fallback `goal-run.json`); the define-goal gate guardrail generalizes to "confirm the goal + gate (measure command OR evaluate goal/rubric)."
- **yaml/schema contract fixes (OR-8, OR-11).** Rewrite `ReuseThresholdSchema`'s comment to occupancy-ceiling language (matching G.1.3 and the "stricter than handoff" note); define `verifyPolicy: standard` and `light` semantics in auto §5.
- **PREAMBLE + LEAD scope (SH-5, SH-7).** Scope the solo "Default to action / investigate proactively" disposition to interactive/standalone sessions (dispatched leaf workers report out-of-scope issues in DONE durable-findings, do not investigate); state the LEAD's explicit exception that it MAY apply trivial inline fixes (Step E.2) though it does not author whole stage artifacts.
- **OPTIONAL parity debt.** Add `ship.ts` / `verify-enhanced.ts` (and `auto`/`goal` command templates if absent) to the parity registry, or relay to child #5 — decided in tasks based on current coverage.
