## Context

`fix-brand-residuals` (archived 2026-07-10) added legacy-marker recognition to the three shell-completion installers so a pre-existing `# OPENSPEC:START/END` block from an older install is found and replaced/removed alongside the current `# RASEN:START/END` block. Re-verifying that fix against the current tree for this change found that every recognition site implements the same shortcut: search the recognized families in a fixed order and act on the **first** one found. That is correct when at most one family's block is present (the only case the predecessor change's tests exercised), but wrong when a profile has blocks under **both** families at once — reconfigure or uninstall then only touches the first-found block and leaves the other orphaned with no future command able to clean it up.

The bug surface turned out to be four independent short-circuit implementations, not the two named when this change was scoped (re-verification against the current tree superseded that count):

1. `src/utils/file-system.ts` `updateFileWithMarkers()` — shared by `bash-installer.ts`'s and `zsh-installer.ts`'s **reconfigure** path. Only searches the legacy pair when the current pair is entirely absent; if the current pair is found, it never checks whether a legacy block also exists elsewhere in the file.
2. `bash-installer.ts` — a private `resolvePresentMarkers()` picks one family (current preferred) and `removeBashrcConfig()`'s line-splice **uninstall** logic acts on that one family only. This path does not go through `file-system.ts` at all.
3. `zsh-installer.ts` — a byte-for-byte duplicate of #2's pattern for zsh's uninstall path.
4. `powershell-installer.ts` `findManagedBlockRange()` — loops the families in order and returns on the first one with both markers present. Used by **both** `configureProfile()` (reconfigure) and `removeProfileConfig()` (uninstall), so PowerShell is affected on both paths.

## Goals / Non-Goals

**Goals:**
- After reconfigure, a profile that had blocks under more than one recognized marker family ends up with exactly one managed block: the first one found, refreshed with current `RASEN` content, at its original position. Every other recognized block is removed.
- After uninstall, a profile that had blocks under more than one recognized marker family ends up with zero managed blocks, regardless of family.
- Existing single-block behavior (fresh install, single-family upgrade, single-family uninstall) is unchanged — the tests added by `fix-brand-residuals` stay green without modification.

**Non-Goals:**
- Consolidating the three independently-duplicated marker-literal definitions (bash/zsh/PowerShell `RASEN`/`OPENSPEC` string constants) into one shared constant. Carried over from `fix-brand-residuals`'s explicit Non-Goal; still true here — this change touches the *search* algorithm, not the literals.
- Touching the exported `removeMarkerBlock()` function in `file-system.ts`. It serves a different, already-retired code path (`legacy-cleanup.ts` for root `CLAUDE.md`/`AGENTS.md` config files) and is unrelated to shell-profile markers.
- Handling more than two marker families, or families with overlapping literal substrings. Only the current `RASEN` and legacy `OPENSPEC` pairs exist today.

## Decisions

**D1 — Extract one shared pure helper (`findAllMarkerBlocks`) for the two character-offset-based consumers; keep the two line-based consumers fixed independently, mirroring the existing duplication.**

Two of the four sites (`file-system.ts`'s `updateFileWithMarkers()` and `powershell-installer.ts`'s block-range search) operate on character offsets into a raw string and need to find *every* recognized block across *all* families, sorted by position, before deciding what to keep and what to drop. That is a non-trivial scan (skip malformed marker state per family without throwing, avoid overlapping matches, handle "family not present at all") — exactly the kind of logic that produced this bug when written independently. A single exported pure function, `findAllMarkerBlocks(content, markerFamilies): MarkerBlockMatch[]`, added to `file-system.ts` and imported by `powershell-installer.ts` (which already imports `FileSystemUtils` from that module — no new cross-layer coupling), replaces two divergent implementations of the same algorithm with one tested one.

The other two sites (`bash-installer.ts`'s and `zsh-installer.ts`'s uninstall paths) work at the line-array level (`content.split('\n')`, `lines.findIndex(...)`), not character offsets, and their removal logic (splice out the line range, trim trailing blank lines) is already correct and already covered by existing tests for the single-block case. Forcing them onto the character-offset helper would mean converting between representations for no benefit. Instead, each keeps its own private "which families are present" check — generalized from returning one preferred family to returning *all* present families — and loops the existing line-splice removal over each. This mirrors the precedent set by `fix-brand-residuals`'s Non-Goal against consolidating the marker-literal constants: bash and zsh are allowed to stay symmetric-but-separate where the mechanism is simple and low-risk; only the genuinely complex offset-scanning logic gets centralized.

Alternative considered: fix all four sites independently in place, no extraction. Rejected — it would mean writing the "scan multiple families, collect every block, handle malformed state" algorithm three times over (file-system.ts, powershell-installer.ts, and implicitly re-deriving similar reasoning for review), which is the exact failure mode that shipped this bug: each site independently took the "prefer current, stop at first match" shortcut. Extraction is the smaller, lower-risk diff here, not a larger one, because the shared piece is a pure function with no I/O or layer-specific concerns (BOM handling, encoding, line-splice) to leak across the boundary.

**D1a — `findAllMarkerBlocks` takes a `strict` option (default `true`); the two consumers do not share one matching strictness.** The two character-offset consumers sharing D1's helper turned out not to share the same marker *format*: bash/zsh's managed block is always a bare marker alone on its line (`# RASEN:START`), while PowerShell's managed block always appends a trailing human-readable description on the start-marker line (`# RASEN:START - Rasen completion (managed block, do not edit manually)`). A single matching rule can't satisfy both — pre-existing bash/zsh matching (via the old `updateFileWithMarkers()`, itself reusing `file-system.ts`'s existing `isMarkerOnOwnLine`/`findMarkerIndex`) requires nothing but whitespace on *both* sides of the marker on its line; PowerShell's old `findManagedBlockRange()` did an unconstrained `content.indexOf()` with no line-position check at all. `findAllMarkerBlocks(content, markerFamilies, { strict? })` resolves this with two matching modes behind one shared scanning algorithm: `strict: true` (the default, used by `updateFileWithMarkers()` for bash/zsh) reuses the existing both-sides `isMarkerOnOwnLine` check unchanged; `strict: false` (used only by `powershell-installer.ts`'s two call sites) uses a new left-side-only `isMarkerAtLineStart` check — the marker must still start its line, but trailing content is tolerated. This was corrected after initial implementation shipped the relaxed check as the *only* mode, which the verifier caught via live repro: `findAllMarkerBlocks` matching a user's own unrelated `.bashrc` comment (`# RASEN:START of my custom aliases` ... `# RASEN:END of my custom aliases`) as a managed block, silently deleting the user's content on `configureBashrc()`. See Risks below.

**D2 — `findAllMarkerBlocks` returns matches sorted by position across all families combined, not grouped by family.** Reconfigure and uninstall both need "the block that appears first in the file," not "the current family's block if present, else the legacy one" — position order is what determines which block is authoritative when both are present. Grouping by family and picking current-first would just reintroduce a milder version of the original bug (guaranteed-safe only when the current-family block happens to be first).

**D3 — Reconfigure algorithm: replace the first match in place, cursor-splice out every other match, collapse resulting blank-line runs.** Naively replacing `blocks[0]`'s span first and then splicing `blocks[1..]` by their original indices is wrong once `blocks[0]`'s replacement text has a different length than the original span (it always does — the legacy marker literal differs in length from `RASEN`'s). The fix builds the result with a single left-to-right pass over the original content using a cursor: copy content up to each match, substitute the first match's span with the fresh managed-block text, drop every subsequent match's span entirely, then copy the remaining tail. A final blank-line collapse (matching the existing `removeMarkerBlock()` cleanup: `/(\r?\n){3,}/g` → `\n\n`) avoids leaving multiple blank lines where a deduped block used to be.

**D4 — Uninstall algorithm: same cursor-splice pass with no first-match substitution — every match's span is dropped.** For `powershell-installer.ts`'s `removeProfileConfig()`, this generalizes the existing single-block removal (which already does a before/after substring split plus trim) to N blocks via the same cursor loop. For bash/zsh, this is the loop-over-present-families change from D1 applied to the existing line-splice code, not a new algorithm.

**D5 — Delta spec: new requirement, not a MODIFY of `Brand namespace identifiers`.** Per the project's ADDED-vs-MODIFY convention, MODIFYING the existing requirement would require reproducing its 3 existing scenarios verbatim inside the delta (risk of accidental drift with no behavior change to justify it). The both-families-present invariant is additive — it doesn't change what the three existing scenarios assert, it adds a case they don't cover — so it goes in a new `### Requirement: Shell-completion marker dedupe on dual-family presence` under `rasen-cli-identity`, with two scenarios (reconfigure dedupes, uninstall clears both).

## Risks / Trade-offs

- [Risk] The cursor-splice rewrite of `updateFileWithMarkers()` and `removeProfileConfig()` changes code that already has passing single-block tests; a subtle off-by-one in the cursor loop could regress those paths. → Mitigation: the single-match case (`blocks.length === 1`) must produce byte-identical output to the current single-block code path before/after refactor — verified by keeping all existing `fix-brand-residuals` regression tests unmodified and green.
- [Risk] Blank-line cleanup after dedup-removal is heuristic (collapse 3+ newlines to 2), which is cosmetic, not exact reproduction of what a user's file looked like before the accidental double-block state existed. → Accepted: the double-block state is itself an already-anomalous condition (the bug this change fixes); exact whitespace fidelity for a state that should never have persisted is not worth extra complexity.
- [Trade-off] `bash-installer.ts` and `zsh-installer.ts` keep independently duplicated "resolve all present families" helpers rather than sharing one. Accepted per D1 — low-risk, low-complexity logic, consistent with the project's existing tolerance for this specific duplication.
- [Risk — caught during verification, fixed] Sharing `findAllMarkerBlocks` between bash/zsh (via `updateFileWithMarkers()`) and PowerShell as a single unconditionally-relaxed matcher weakened bash/zsh's own matching criteria from "marker alone on its own line" to "marker starts the line, anything can follow." A `.bashrc`/`.zshrc` containing a user's own comment that happens to line-start with the marker literal (e.g. a section-header convention unrelated to this tool) would be matched as a managed block and have its content silently deleted on reconfigure. → Fixed per D1a: `findAllMarkerBlocks` takes a `strict` option, defaulting to the pre-existing strict both-sides check; only PowerShell's two call sites opt into `strict: false`. The bash/zsh line-splice uninstall path (`resolveAllPresentMarkers` + `lines.findIndex((line) => line.trim() === markers.start)`) was independently audited and confirmed *not* exposed to the same issue — its match is always an exact full-line-after-trim equality check, stricter than either `findAllMarkerBlocks` mode, and predates this change. Regression tests added: `updateFileWithMarkers()`, `configureBashrc()`/`removeBashrcConfig()`, and `configureZshrc()`/`removeZshrcConfig()` each get a coincidental-marker-prefix fixture proving the user's content survives untouched.

## Migration Plan

No runtime migration. This is a pure bug fix to code paths that only run when a user invokes `rasen completion install`/`configure`/`uninstall`. The next time any of those commands runs against a profile with orphaned dual-family blocks, the orphan is cleaned up automatically — no version gate or data migration needed.
