## Judgment precedents established (apply consistently to remaining files)

1. **Brand-word scope vs. CLI-accuracy scope.** Where EN's content structurally diverges from ZH for reasons unrelated to branding (e.g. `supported-tools.md`'s Skills/Commands two-column table вҖ” EN dropped the Commands column because delivery went skills-only; `cli.md`'s `rasen config profile` вҶ’ `rasen profile` rename), fix ONLY the brand word/literal identifier inside the stale sentence, don't restructure content to match EN's newer facts. That's the sibling change `fix-docs-broken-tables`'s job ("Documented commands match the shipped CLI"). Note these as brand-unrelated staleness in your own return; don't silently fix or silently leave unremarked.
2. **Historical-narrative files can be "under-branded" in the wrong direction.** For the four category-1 files (`upstream-v1.5-stores-and-resolution.md`, `review-cycle-workflow-design.md`, `codex-workflow-integration.md`, `grill-gstack-absorption.md`), don't assume more "OpenSpec"/pre-rebrand-syntax = residual. Check whether EN's own count/positions match ZH's вҖ” twice now (review-cycle-workflow-design.md, grill-gstack-absorption.md) I found ZH had *fewer* occurrences than EN because an earlier pass over-corrected literal historical commands (`/opsx:auto`, `openspec init`, `openspec/changes/`) to modern `/rasen-*` form. The fix in that direction is to REVERT to the historical literal, not "further modernize." Always diff occurrence counts/positions against EN first; don't touch these files blind.
3. **Fast screening technique.** For any file, `grep -nio "openspec" docs/<f>.md | cut -d: -f1` vs the same on `docs/zh/<f>.md` вҖ” if the position lists (or at minimum occurrence counts) match, the file is very likely already aligned and doesn't need a full read/edit pass; spot-check 2-3 positions to confirm before skipping. Several files (local-install.md, codex-workflow-integration.md, upstream-v1.5-stores-and-resolution.md) turned out to need zero edits this way. For the colon-syntax historical files, also separately grep `opsx:` (EN) vs `rasen-` (ZH candidates-for-revert) since colon-command mismatches don't contain the literal substring "openspec" and won't show up in the openspec-only grep.
4. **`.openspec.yaml` and other still-current literal filenames stay as `openspec`.** Confirmed via `package.json`'s `bin` field that the real CLI binary is `rasen`, but some literal artifacts genuinely are still named with the `openspec` prefix today (e.g. `.openspec.yaml` change-metadata files вҖ” verified against a real file in this very change directory). Don't assume every literal `openspec` string is stale; check against real repo state when in doubt.
