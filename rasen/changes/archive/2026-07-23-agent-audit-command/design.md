## Context

`scripts/token-audit/audit.mjs` (committed at `2e8639ee`) is a working, validated session-cost analyzer: it streams a Claude Code main transcript plus its subagent transcripts, dedupes usage by `message.id` (line-counting overstates spend ~2.5x), applies two cache-write TTL coefficients (main session 1h/2x, subagent 5m/1.25x), classifies cache-churn events by cause, and emits a self-describing `rasen-token-audit/1` JSON that `viewer.html` (a self-contained, dependency-free single file) renders as an interactive timeline. It was deliberately kept out of the CLI surface because it parses Claude Code's undocumented transcript format, which can drift across harness versions without notice.

This change productizes that tool as `rasen agent audit`, in the existing `rasen agent` namespace (`src/commands/agent.ts` — currently `context` and `wait`, both thin consumers of logic living in `src/core/`). The command must reuse the Claude-projects discovery already implemented for `agent context` (`claudeProjectsDir`, `findLatestMainTranscript` in `src/core/agent-context.ts`) rather than re-deriving the project-directory slug, and it must expose the drift risk instead of hiding it: an experimental marker plus fail-soft error handling replace "keep it out of the CLI entirely" as the mitigation.

**Scope addition — Codex rollout support (added after initial design pass).** `agent context` already supports probing a Codex CLI rollout via `--runtime codex`, backed by a substantial existing reader layer in `src/core/codex/` (`rollout.ts`: `findRolloutPath`, `listRolloutFiles`, `readRolloutOccupancy`, `readRolloutConversation`; `codex-home.ts`: `resolveCodexHome`). `rasen agent audit` must offer the same `--runtime codex` capability rather than being Claude-only, reusing that reader layer rather than writing a second Codex parser. A real rollout sample (`~/.codex/sessions/2026/07/23/rollout-...-019f8bae....jsonl`) and the repo's own `test/fixtures/codex-rollout/sample-rollout.jsonl` confirm the shape used below: each line is `{timestamp, type, payload}`; `session_meta` (always the first line) carries `cwd`, `thread_source` (`"user"` for a main thread, `"subagent"` for a spawned one), `parent_thread_id`/`forked_from_id` (subagent lineage), `agent_nickname`/`agent_path`, and `cli_version`; `event_msg` rows with `payload.type: "token_count"` carry `info.total_token_usage` (a monotonic CUMULATIVE counter) and `info.last_token_usage` (the state as of that event — repeated across stream-update re-emissions, not a clean per-request delta) plus `info.model_context_window`; `event_msg` rows with `payload.type: "task_started"`/`"task_complete"` (each carrying a `turn_id`) bound a turn.

## Goals / Non-Goals

**Goals:**
- Ship `rasen agent audit <sessionId|path>` as a documented, experimental CLI command that a user runs locally, with output landing on their own machine (`~/.rasen/analytics/`) and no network calls.
- Preserve the analyzer's measurement discipline exactly (message.id dedup, dual TTL pricing, churn cause attribution) — this is a migration of proven logic, not a rewrite.
- Support Codex CLI rollouts via `--runtime codex`, symmetric with `agent context`, reusing the existing `src/core/codex/` reader layer rather than duplicating it.
- Make format drift fail loud-but-friendly for the user and loud-and-caught for CI (fixture test), never an unhandled exception — for both runtimes.
- Ship `viewer.html` as a distributable package asset with a `--open` convenience flag, capable of rendering a report from either runtime.
- Ship a `/rasen-audit` guidance skill so a user who doesn't know the command exists can still get to an audit, on either runtime.

**Non-Goals:**
- No hook-based automatic collection (`--install-hook`, Stop/SessionEnd hooks) — that's a pull-only tool in this phase.
- No upload, telemetry contract changes, or consent tiering — `src/telemetry/` is untouched; the audit's own invocation goes through the existing command-name telemetry unmodified (same as any other command), and that's the only telemetry footprint this change has.
- No rich local JSONL event log / gstack-style event stream.
- No version bump — this is a normal feature addition, not a release action.
- No dollar-denominated cost estimate for Codex sessions, and no attempt to reconcile Codex's and Claude's accounting into one unified "cost" number — see D6.

## Decisions

### D1 — Where the audit logic lives, and what happens to `audit.mjs`

New module: `src/core/token-audit/` (mirrors `src/core/agent-context.ts`'s role for `agent context`), split as:
- `src/core/token-audit/parse.ts` — per-file streaming parse of a **Claude** transcript (dedup by `message.id`, tool census) — ported near-verbatim from `audit.mjs`'s `parseFile`.
- `src/core/token-audit/classify.ts` — churn classification (`classify`) and burst clustering (`clusterBursts`), pricing constants — Claude-only (see D6 for why this has no Codex counterpart).
- `src/core/token-audit/parse-codex.ts` — per-thread parse of a **Codex** rollout: per-request delta derivation and turn segmentation (D5), built on `src/core/codex/rollout.ts`'s existing row-reading primitives rather than a second JSONL reader.
- `src/core/token-audit/discover-codex.ts` — Codex subagent-family discovery (D5), built on `listRolloutFiles`/`findRolloutPath` from `src/core/codex/index.ts`.
- `src/core/token-audit/audit.ts` — orchestration (`runAudit`): resolves the runtime (explicit `--runtime`, else filename/content detection reusing `detectTranscriptKind` from `src/core/agent-context.ts`), discovers files for that runtime, runs the matching parse path, aggregates, sorts agents into activation order, and builds the runtime-appropriate result object (D6). This is what `AgentCommand.audit()` calls.
- `src/core/token-audit/errors.ts` — the fail-soft error taxonomy (see D3), shared by both runtime parsers.

`scripts/token-audit/audit.mjs` becomes a **thin deprecated wrapper** that shells out to `rasen agent audit` (via `node bin/rasen.js agent audit ...` or equivalent) rather than being deleted. Rationale over outright removal: the script is referenced by name in `rasen/office-hours/token-cost-audit.md` (a durable research record) and by an already-run baseline validation (`README.md`'s "Measurement discipline" section cites session `c4a16986` results the script itself produced) — keeping the file present as a redirect preserves that provenance trail without a broken path, while a one-line deprecation notice in `scripts/token-audit/README.md` points readers to the new command. `forensics/` and `README.md` are untouched otherwise per the proposal's scope.

Alternative considered: delete `audit.mjs`, rewrite `README.md` to point at the new command only. Rejected because it breaks the direct `node scripts/token-audit/audit.mjs <session>` invocations already documented in prior office-hours research notes and offers no benefit over a wrapper (both require README edits either way; the wrapper additionally keeps working for anyone who copy-pastes the old command from memory).

### D2 — `viewer.html` distribution and `--open`

`viewer.html` moves to a new top-level package directory, `assets/token-audit/viewer.html` is rejected (the existing `assets/` dir is doc-images only and isn't shipped — mixing shipped and unshipped content under the same directory name invites a packaging mistake). Instead: `viewer/audit.html` at the repo root, added as a new entry in `package.json`'s `files` array (`"viewer"`, alongside the existing `dist`, `bin`, `schemas`, `skills`, `pipelines` entries). At runtime, the command resolves the shipped copy's path the same way other package-relative assets are resolved elsewhere in the codebase (`createRequire(import.meta.url)` + a path relative to the resolved `package.json`, as `src/commands/ui-launch.ts` and others already do) — never a path relative to `process.cwd()`.

`--open` reuses the existing cross-platform "open in default browser" pattern already implemented in `src/commands/ui-launch.ts` (`openInBrowser`: `open` on darwin, `cmd /c start` on win32, `xdg-open` elsewhere; spawned detached, stdio ignored, `unref()`'d so it never holds the CLI process open, errors swallowed since the URL/path is already printed for manual opening). `rasen agent audit --open` prints the generated JSON's path, then opens `viewer/audit.html` with a `?src=<file-url-to-json>` query — `viewer.html` already supports `?src=<url>&theme=...` when served over HTTP; for a local `file://` open we instead pass the JSON path and have the viewer's existing drag-drop load path double as a `?src=` file read (same code path, no new viewer logic — `file://` fetches work in Chrome/Edge/Firefox for a same-directory or descendant file, which the generated JSON is by default since it lands in the same `~/.rasen/analytics/` tree the command controls).

Alternative considered: copy `viewer.html` next to the generated JSON on every run (like Playwright's HTML report). Rejected for Phase 1 — it duplicates a static asset per invocation for no benefit when the packaged copy is already reachable by relative `file://` link, and it complicates the "delete `~/.rasen/analytics/` any time" story from the proposal (a copied-in viewer would need its own cleanup story).

### D3 — Fail-soft format-drift handling

A single typed error, `TranscriptFormatError`, thrown by the parse layer whenever an assumption about the transcript shape breaks in a way that would silently corrupt the accounting math — as opposed to a benign absence, which is skipped, matching `audit.mjs`'s own `if (u && sum > 0)` guard verbatim (post-review correction, see M2 in `rasen/changes/agent-audit-command/work/review-report.md`; the original script silently skips any assistant line whose `message.usage` is absent, empty, or all-zero, treating it as non-billing — that is NOT drift, and the migration must preserve it exactly, per the Goals section's "migration of proven logic, not a rewrite"). Concretely: an assistant entry with no `message` field, or with `message.usage` entirely absent/non-object, is skipped exactly like a plain unparseable JSON line. The ONE throw trigger is a `message.usage` object that IS present but carries a **non-numeric value** for a token field the pricing math reads (e.g. `output_tokens: "nine"`) — that is the one shape violation `|| 0`-style coercion cannot absorb without silently corrupting a total, so it aborts that file with `TranscriptFormatError` carrying the file path, line number, and a short cause string instead of miscounting. The parser does NOT attempt partial recovery mid-file once that throw fires.

`AgentCommand.audit()` catches `TranscriptFormatError` (and only that type — any other thrown error is a genuine bug and surfaces normally) and prints: `"rasen agent audit: transcript format not recognized (harness may have updated the session log format). <cause> at <file>:<line>. This command is experimental; see rasen/... for details."` with exit code 1 — never a raw stack trace. `--json` mode emits `{"available": false, "reason": "format-drift", "detail": ...}` mirroring `agent context`'s established unavailable-shape convention (see `cli-agent-context` spec) so scripted callers get a stable machine-readable shape instead of parsing prose.

A fixture regression test (`test/fixtures/token-audit/`) pins this: one minimal valid synthetic transcript (few lines, covering message.id dedup — a repeated id with a higher `output_tokens` on the second line — and both TTL tiers via one main + one subagent file) proves normal-path correctness against hand-computed expected totals; a **non-numeric token field** variant proves the one real throw trigger fires (`TranscriptFormatError`, friendly message, not a raw exception); and two **skip-and-continue** variants — `message.usage` entirely absent (`skip-missing-usage/`, M2) and a truncated/unparseable JSON line (`malformed-truncated/`) — prove those benign cases do NOT abort the file, matching the original script's tolerance. A parallel Codex fixture set under `test/fixtures/token-audit/codex/` (synthetic, modeled on `test/fixtures/codex-rollout/sample-rollout.jsonl`'s confirmed shape — never the real sample path the LEAD verified against) covers the equivalent classes: a valid multi-turn rollout plus a subagent rollout, a throwing variant (missing `info.total_token_usage`, or a non-numeric token field — Codex's `token_count` event has no original-script precedent for a silent-skip-on-absent case the way Claude's per-line usage does, so this boundary is unchanged from the original implementation), and a truncated-line skip-and-continue variant. None of the fixtures — Claude or Codex — contain real session data or real local paths; every field is synthetic.

### D4 — `~/.rasen/analytics/` output directory resolution

The default output directory is `path.join(getGlobalDataDir(), 'analytics')`, reusing `getGlobalDataDir()` from `src/core/global-config.ts` — the same machine-data-root resolver already used for the project registry, store registry, and workset state (`RASEN_HOME` > `XDG_DATA_HOME/rasen` compatibility alias > `~/.rasen` default). This makes `rasen agent audit`'s output location follow the same relocation/override rules as the rest of the tool's machine state instead of inventing a second convention. A `--out <path>` flag (mirroring `audit.mjs`'s existing flag) overrides the destination file directly, matching the discovery precedent set by `agent context`'s `--dir`/`--transcript` overrides. This resolution is runtime-agnostic — the same output directory and `--out` override apply whether the analyzed session is Claude or Codex.

### D5 — Codex rollout discovery and per-request delta derivation

**Runtime selection** mirrors `agent context` exactly: an explicit `--runtime codex` wins outright; otherwise the target is detected from its filename (`rollout-*.jsonl`, `CODEX_ROLLOUT_BASENAME` in `src/core/agent-context.ts`) or first-line content sniff (`detectTranscriptKind`) when a direct path is given. A bare session/thread id has no runtime-identifying shape of its own (both runtimes key by UUID), so **resolving a bare id to a Codex thread requires `--runtime codex`** — without it, a bare id is resolved as a Claude session id (existing behavior, unchanged default). When `--runtime codex` is given, the target id is resolved to a rollout path via the existing `findRolloutPath(threadId, { codexHome })` (`src/core/codex/rollout.ts`) — the same bounded newest-first active-then-archived scan `agent context` already relies on — not a new lookup.

**Subagent discovery** has no `subagents/` subdirectory to scan (unlike Claude). Instead: `listRolloutFiles(sessionsDir)` (already exported, bounded to the fixed `sessions/<Y>/<M>/<D>/` tree) enumerates every rollout, each candidate's `session_meta` first line is read for `parent_thread_id`, and a BFS over that parent-child relation starting at the target thread id collects its full subagent family (transitively — a subagent's own subagent still belongs to the audit). `agent_nickname` (falling back to `agent_path`, falling back to the thread id) becomes that rollout's agent label, the Codex analog of `audit.mjs`'s `parseAgentLabel`. **Implementation note**: the first-line `session_meta` reader currently lives as a private, unexported `readSessionMeta` inside `src/core/agent-context.ts` (used there for `findLatestRollout`'s cwd/non-forked filtering) — this change relocates it to `src/core/codex/rollout.ts` as an exported `readRolloutSessionMeta` (function body unchanged, same signature/behavior) and updates `agent-context.ts` to import it from there. This keeps exactly one `session_meta`-reading implementation instead of two, which is what "reuse `src/core/codex` rollout parsing" requires — a low-risk rename-and-relocate, not new logic.

**Per-request delta derivation — the Codex analog of Claude's message.id dedup.** A rollout's `token_count` events fire on every stream update (163 in the LEAD's 258-line sample), and `info.last_token_usage` is the state as of that event, not a clean per-request delta — the same event's numbers can repeat verbatim across several consecutive lines before the next turn's numbers appear. The rule: track `info.total_token_usage.total_tokens` (and its sibling fields) as a monotonic cumulative counter; each `token_count` event whose cumulative total differs from the last-recorded value contributes exactly one derived request record, with per-field deltas computed against the previous recorded cumulative snapshot; an event whose cumulative total is unchanged from the last recorded one is skipped (it is a re-emission of the same state, the Codex counterpart of a repeated `message.id` line). The first recorded event's delta is the raw cumulative value itself (nothing preceded it). **Turn boundaries** come from `task_started`/`task_complete` `event_msg` rows (each carrying `turn_id`, already the vocabulary `readRolloutConversation`'s `turnIds` extraction uses) — each derived request is attributed to the turn whose `task_started`/`task_complete` window contains its timestamp, giving a `turns[]` structure analogous to Claude's `bursts` for the viewer's timeline, without borrowing Claude's warm/cold HIT/MISS vocabulary (D6: there is no cache-TTL model to classify a "miss" against).

**Fork/resume handling — flag, do not exclude (M1 fix, post-review).** `session_meta.forked_from_id` marks a rollout that replays another session's history into its own file — Codex's `resume`/fork workflow is a normal way to continue a session, not a rare edge case. When set, every `token_count` transition recorded during that replay is indistinguishable from a real per-request delta under the rule above, at whatever (compressed) timestamps the replay was logged — so request counts, turn timings, and `cacheHitRatio` computed before the thread's own first local turn are not per-request trustworthy. This is deliberately NOT solved by excluding replayed rows: a heuristic boundary (e.g. "everything before the first local `task_started`") is a real fix but unvalidated guesswork today, and a silently-adjusted number is worse than an honestly-flagged one — that refinement is future work. Instead: `discoverCodexThreadFamily`/`buildCodexFamilyMember` (`src/core/token-audit/discover-codex.ts`) read `forked_from_id` into `CodexFamilyMember.forkedFromId`; when the AUDITED (target) thread's own `forkedFromId` is set, `runCodexAudit` sets `session.forkedFrom` on the report and appends one entry to a `caveats: string[]` array (schema stays additive, `rasen-token-audit/2`) naming the parent thread and what is not trustworthy; the CLI's text-mode summary prints each caveat as a `Caveat: ...` line, and the viewer renders a warning banner when `caveats` is non-empty — cheap, runtime-agnostic surfacing, no new render logic per runtime.

**Asymmetry with `agent context`'s `--latest` discovery, explained.** `agent-context.ts:findLatestRollout` (used by `agent context --latest`) already excludes any candidate with `forked_from_id !== undefined` from consideration — a forked rollout should never be silently adopted as "the" newest session for a cwd. `rasen agent audit` inherits that exclusion for free wherever it reuses `findLatestRollout`, but **`agent audit` has no `--latest` flag at all** — every invocation names its target explicitly, either a bare thread id (resolved via `findRolloutPath`, which does not filter on `forked_from_id`) or a direct rollout path. So the exclusion that protects `--latest` discovery never applies to audit's actual resolution path, which is exactly why the flag (not a reused exclusion) is the correct mechanism here: an explicitly-named fork is a legitimate, deliberate audit target (the user asked for that thread specifically), so it must be analyzed, just honestly labeled.

### D6 — Codex cache/pricing model divergence and report schema version

Codex's accounting is not a drop-in substitute for Claude's: `cache_write_input_tokens` is typically 0 (OpenAI's caching is automatic, not a billed write the way Claude's cache-creation tokens are), there is no TTL-tiered pricing to apply, and `reasoning_output_tokens` is a field Claude's usage object has no equivalent for. Two consequences:

1. **No `billedInputEq` for Codex.** Claude's "billed-input-equivalent" is explicitly an Anthropic-cache-economics heuristic (`raw + write×TTL-coefficient + 0.1×read`), not a real dollar figure even for Claude — inventing an analogous single number for Codex without a validated formula would fabricate false precision. A Codex report instead exposes raw component totals per agent/turn (`inputTokens`, `cachedInputTokens`, `cacheWriteInputTokens`, `outputTokens`, `reasoningOutputTokens`, `totalTokens`) and a `cacheHitRatio` (`cachedInputTokens / inputTokens`) as a health signal — informative without pretending to be a cost.
2. **No churn-cause taxonomy for Codex.** `ttl-expiry`/`rebase`/`context-drop` are classifications against Claude's specific TTL/prefix-match cache model; Codex's automatic caching has no equivalent mental model to classify a "miss" against. The Codex report's `turns[]` (D5) gives timeline structure without asserting a cause for any given cache-hit-ratio dip.

**Schema version**: bump to `rasen-token-audit/2` (additive over `/1`, not a breaking rename) rather than force-fitting Codex data through the `/1` shape with absent/null Claude-only fields. `/2` adds `session.runtime: 'claude' | 'codex'` (defaulting the reader's interpretation) as the discriminant; a `/1` (Claude-only, no `runtime` field) or a `/2` report with `runtime: 'claude'` both carry the full original shape (`pricing`, `churnEvents`, `resumes`, `billedInputEq`) unchanged; a `/2` report with `runtime: 'codex'` omits those Claude-specific fields entirely and adds `rawTokens`/`cacheHitRatio`/`turns[]` instead, rather than padding them with misleading zeros. Rationale for a version bump over silently overloading `/1`: a consumer (the viewer, or any future scripted reader) that branches on `runtime` needs to know unambiguously that it must NOT assume `pricing`/`churnEvents` exist — a version bump makes that contract explicit and machine-checkable (`schema !== 'rasen-token-audit/1'` is a cheap guard) instead of relying on every consumer to defensively feature-detect every field. `viewer.html` gains a runtime-aware render branch (task item) rather than assuming Claude-only fields are always present.

Alternative considered: unify Codex's `reasoningOutputTokens` into Claude's `outputTokens` bucket and force a `billedInputEq` estimate using Codex's list-price input/output/cache-read rates (no TTL tiers needed since Codex has none). Rejected for Phase 1 — Codex's actual list pricing varies by model and by provider routing (`model_provider` in `session_meta` was `"proxy"` in the live sample, not directly `"openai"`), so a hardcoded rate table would be guessing at a number the user could mistake for their real bill; raw totals are honest about what is and isn't known.

## Risks / Trade-offs

- **[Risk] Transcript format drifts and the fixture doesn't cover the new shape** → Mitigation: fixtures are small and cheap to extend; the experimental marker in both the command's own output and the skill sets user expectations up front, and the fail-soft path (D3) means drift degrades to a clear error rather than a confusing crash even before a fixture is updated — this applies identically to a Codex rollout shape drift (e.g. a future `token_count` payload restructuring).
- **[Risk] `file://` viewer load is blocked by a browser's local-file security policy on some platforms/profiles** → Mitigation: the command always prints the raw JSON path as a fallback instruction ("drag this file onto `viewer/audit.html`"), so `--open` failing to auto-load never blocks getting the data.
- **[Risk] Packaging a new `viewer` directory into `files` bloats the npm tarball** → Mitigation: `viewer.html` is already a small (single-digit KB) self-contained file with no assets of its own; negligible compared to `dist`.
- **[Trade-off] Keeping `audit.mjs` as a wrapper instead of deleting it** means two entry points exist for a while → acceptable: the wrapper is a redirect, not a maintained second implementation, and it's a one-time cost that protects prior research provenance.
- **[Risk] Codex subagent BFS scan reads every rollout's first line across the whole `sessions/` tree for a session with many candidates** → Mitigation: bounded by the same fixed three-level date-directory depth `listRolloutFiles` already bounds `agent context`'s discovery by; each read is a first-line-only sniff (not a full-file parse), and this is a one-shot pull-model command, not something run per-request in a hot path.
- **[Trade-off] `rasen-token-audit/2`'s Codex shape and Claude shape share a schema name but not a field set** → acceptable and disclosed: the `runtime` discriminant makes the divergence explicit and checkable rather than hidden; alternative (two entirely separate schema names) was rejected as needless ceremony for what is still fundamentally "one session's token audit."
