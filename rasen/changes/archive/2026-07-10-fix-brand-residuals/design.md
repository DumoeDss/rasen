## Context

An independent brand sweep after `specs-brand-rewrite` (archived 2026-07-10) found three residual `OpenSpec` tokens (A1/A2/B1, see `planning-context.md`) plus flagged two behavioral questions that had to be answered against the actual code before deciding whether the fix is a cosmetic token swap or a real behavior-description correction: F5 (does any marker-scoped partial-update mechanism survive in the current update path?) and F6 (do the shell-completion installers actually write the legacy marker into new content, violating `rasen-cli-identity/spec.md:59`?). This design records the investigation evidence and the resulting fix shape.

## Investigation: F5 — is the marker-scoped partial-update mechanism real?

**Verdict: no. It is uniformly stale across all 8 sites in `cli-update/spec.md` (lines 24, 30, 59, 87, 112, 118, 124, 131 — CodeBuddy, Factory Droid, Codex, GitHub Copilot, Gemini CLI, iFlow, plus the two generic "OpenSpec-managed" requirement lines). There is no per-tool nuance — every AI-tool adapter does whole-file regeneration, full stop.**

Evidence:

- `src/core/update.ts:224` (skill files) and `src/core/update.ts:252` (command files) both write via `FileSystemUtils.writeFile(path, content)` — a whole-file overwrite, never a marker-scoped splice.
- `src/core/command-generation/adapters/{codebuddy,factory,codex,github-copilot,gemini,iflow}.ts` — read directly, all six. Every adapter's `formatFile()` builds file content purely from the fresh `CommandContent` template parameter. None reads the existing on-disk file. None reference any marker constant. None preserve "unmanaged content outside markers" — there is nothing in the code path that could preserve it, because the existing file is never read before being overwritten.
- `src/core/config.ts:14-15` — the code comment on `OPENSPEC_MARKERS` states explicitly: "Legacy marker pair — recognized ONLY to identify pre-existing legacy artifacts (upstream OpenSpec or older rasen installs). Never written into new content." This constant is used only by `legacy-cleanup.ts` for detecting/removing markers from **root shared config files** (`CLAUDE.md`, `AGENTS.md`, etc. — the old `ToolRegistry`-era config file mechanism), which is a different, already-retired code path from command-file generation and out of scope for this change.

Conclusion: the 8 `cli-update` scenarios describe a "replace only the content between markers, preserve everything else" mechanism that does not exist anywhere in the current command-generation pipeline for any tool. This is a real behavior-description defect, not a brand-token issue — a silent `OpenSpec` → `Rasen` swap on these lines would leave the spec describing behavior that never happens. The fix rewrites these scenarios to describe what the code actually does: full regeneration of each command file from the current shared template on every `rasen update` run, with the "preserve user customizations" framing removed where it isn't true (whole-file overwrite has no place to preserve anything) and kept only where a scenario is genuinely about a different mechanism (e.g. Gemini CLI's TOML `prompt = """` framing preservation, which the adapter *does* do structurally — see Decisions below).

**Separate finding, explicitly out of scope**: the same spec file's root-level `AGENTS.md`/`CLAUDE.md` "stub" scenarios (roughly lines 10-13, 18-21, 26-29, 36-39 — "if a root-level stub exists, refresh it so it points to `@/rasen/AGENTS.md`") describe a mechanism that a repo-wide search (`@/rasen/AGENTS.md`, `@/openspec/AGENTS.md`, "root-level stub", "rootStub", "ensureRootStub") found zero matches for anywhere in `src/`. This looks like a larger, independent staleness issue (a described feature with no implementation at all, vs. this change's scope of a described mechanism that's merely the wrong shape). Flagged as a follow-up; not touched here per the scope guard against pulling in undiscovered ledger items.

## Investigation: F6 — do the shell installers violate the marker-write prohibition?

**Verdict: yes, confirmed in three independent places.** `rasen-cli-identity/spec.md:59` states: "The legacy marker pair `<!-- OPENSPEC:START -->` / `<!-- OPENSPEC:END -->` SHALL remain recognized solely for identifying legacy artifacts, and SHALL NOT be written into newly generated content." (The installers use a shell-comment variant, `# OPENSPEC:START` / `# OPENSPEC:END`, but it's the same `OPENSPEC` token being newly written, which is the substance of the requirement.)

Evidence:

1. `src/core/completions/installers/bash-installer.ts:17-20` defines its own `BASHRC_MARKERS = { start: '# OPENSPEC:START', end: '# OPENSPEC:END' }` — an independent literal, not imported from `src/core/config.ts`'s `OPENSPEC_MARKERS`. `configureBashrc()` (~line 136) calls `FileSystemUtils.updateFileWithMarkers(bashrcPath, config, BASHRC_MARKERS.start, BASHRC_MARKERS.end)`.
2. `src/utils/file-system.ts:240-276`, `updateFileWithMarkers()` — the shared helper both bash and zsh installers route through. When no existing markers are found (fresh `.bashrc`/`.zshrc`, or first install), it writes `startMarker + '\n' + content + '\n' + endMarker` (lines 267 and 272) — i.e., it writes the literal `# OPENSPEC:START`/`# OPENSPEC:END` text into brand-new content whenever whatever markers were passed in aren't already present.
3. `src/core/completions/installers/zsh-installer.ts:17-20` — an identical, independently duplicated `ZSHRC_MARKERS` literal, hitting the same violation through the same shared helper.
4. `src/core/completions/installers/powershell-installer.ts:217-223` — does not use the shared helper at all. `configureProfile()` inlines its own block construction and hardcodes `'# OPENSPEC:START - Rasen completion (managed block, do not edit manually)'` / `'# OPENSPEC:END'` directly. A third, independent implementation of the same marker literal. `removeProfileConfig()` (~lines 270-271) independently hardcodes the same two marker strings again for uninstall-time detection.

Also confirmed: the marker pair is defined **three separate times** (bash, zsh, PowerShell) with no shared constant — real duplication, noted but not refactored away (see Non-Goals).

Existing test coverage: `test/core/completions/installers/{bash,zsh,powershell}-installer.test.ts` already assert `# OPENSPEC:START`/`# OPENSPEC:END` literally, including on fresh-install paths (e.g. `bash-installer.test.ts:118-119`) — these assertions currently encode the bug and must flip to assert `RASEN` on new writes. Other existing tests (e.g. `zsh-installer.test.ts:598,607,732`, `powershell-installer.test.ts:386-388`) seed a pre-existing legacy `OPENSPEC` block to test detection/replacement/removal — those seed fixtures stay `OPENSPEC` (that's the point: simulating an old install) but their result assertions need to reflect the new dual-recognition behavior.

## Goals / Non-Goals

**Goals:**
- Correct the 8 `cli-update` scenarios (plus the 2 generic requirement lines) to describe actual whole-file-regeneration behavior instead of a fictional marker-scoped partial update.
- Stop bash/zsh/PowerShell completion installers from writing the legacy `OPENSPEC` marker into brand-new shell-profile content; they write `RASEN` markers going forward.
- Preserve backward compatibility: a profile with an existing legacy `OPENSPEC` block is still recognized, replaced in place (no duplicate block) on re-configure, and removable on uninstall.
- Fix the `isOpenSpecProject` token in `cli-completion/spec.md` and the `OpenSpec` word in `config.yaml` — plain conformance edits, no delta needed.

**Non-Goals:**
- Consolidating the three independent marker-literal definitions (bash/zsh/PowerShell) into one shared constant. It's real duplication, but collapsing it isn't necessary to fix the violation and would expand the diff surface across three files' internal structure for no behavior change. Flagged, not refactored.
- Fixing the root-`AGENTS.md`/`CLAUDE.md` stub staleness in `cli-update/spec.md` (separate, larger issue — see F5 finding above).
- Pulling in any F1-F4 ledger item from `specs-brand-rewrite`, or touching historical `explorations/`/`office-hours` docs.
- Any change to `rasen update`/`rasen init` *behavior* beyond the corrected spec wording — no new preservation logic is being added to command-file generation, because none exists today and the spec correction is what's changing, not the code.
- Changing what `isOpenSpecProject`/`isRasenProject` does in code — it's a fictional interface-sketch method in both the old and new spec text; only the token changes.

## Decisions

**D0 — fix scope within `cli-update` extends 3 lines beyond the originally-flagged 8 sites, for internal consistency.** While reconstructing the full `Slash Command Updates` requirement block for the MODIFIED delta (required — the workflow mandates copying the entire requirement, not just the changed scenarios), three more scenarios in the *same* requirement were found to make the identical false claim that the sweep's 8-site list didn't catch: Antigravity ("refresh the OpenSpec-managed portion of each file", ~line 47) and Windsurf/Kilo Code ("refresh each file using shared templates wrapped in Rasen markers", ~lines 98/104). Verified directly against `antigravity.ts`, `windsurf.ts`, `kilocode.ts` — none of the three does any marker wrapping; all three are plain whole-file `formatFile()` regeneration, same as the other 6. Leaving these 3 scenarios describing a marker mechanism while fixing their siblings in the same requirement block would reintroduce the exact inconsistency this change exists to remove, so they're corrected too. This does not expand delta count or touch any other capability — same MODIFIED delta, same requirement block, already being fully rewritten.

**D1 — `cli-update` fix is a MODIFIED delta, not a silent token swap.** Per the F5 evidence, the marker-scoped mechanism is false for every tool, not merely mis-branded. A token-only `OpenSpec` → `Rasen` swap would leave the spec asserting a still-false mechanism ("Rasen managed markers" instead of "OpenSpec managed markers") under a corrected name. The delta rewrites the affected scenarios' `AND` clauses to state whole-file regeneration plus (where genuinely true) structural-framing preservation, e.g.:
  - Generic requirement ("Tool-Agnostic Updates", line ~24/30): replace "refresh OpenSpec-managed files" / "update only the OpenSpec-managed sections... leaving user-authored content untouched" with language describing full regeneration of the managed command/skill file from the current template on every update.
  - CodeBuddy (line 59), Factory Droid (87), Codex (112), GitHub Copilot (118), iFlow (131): replace "preserve...outside the OpenSpec managed markers" / "update only the content inside the OpenSpec managed markers" with "regenerate the file from the current template" — dropping the marker-preservation framing since `formatFile()` has no user-editable region to preserve; frontmatter (`description`, `argument-hint`, etc.) is regenerated fresh each time, which is what the adapters actually do.
  - Gemini CLI (line 124): this one is the partial exception — the adapter *does* wrap the regenerated body inside a fixed TOML `description = "..." / prompt = """ ... """` structural frame (see `gemini.ts` `formatFile()`), so the corrected scenario keeps a "preserve TOML framing, regenerate the body" framing, but drops the literal `<!-- OPENSPEC:START -->`/`<!-- OPENSPEC:END -->` marker-syntax claim (that literal string does not appear anywhere in `gemini.ts`).

**D2 — F6 fix: dual-marker recognition, write current marker only.** `updateFileWithMarkers()` in `file-system.ts` and the PowerShell installer's inline block logic both need to: (a) search for an existing block under either the legacy `# OPENSPEC:START`/`# OPENSPEC:END` pair or the current `# RASEN:START`/`# RASEN:END` pair when deciding whether to replace-in-place vs. append-new; (b) always **write** the current `RASEN` pair, regardless of which pair was found (or none). Concretely:
  - `FileSystemUtils.updateFileWithMarkers()` gains an optional way to recognize a legacy marker pair in addition to the pair being written — simplest shape: accept an additional `legacyStartMarker`/`legacyEndMarker` pair (or a small ordered list of recognized pairs) and search for any of them; when found, replace that span; when writing, always use the passed-in (current) `startMarker`/`endMarker`. This keeps the existing signature's primary contract (write X) while adding legacy lookup, so bash/zsh installers pass `RASEN` markers to write and `OPENSPEC` markers to also recognize.
  - `powershell-installer.ts`'s `configureProfile()` block-construction and `removeProfileConfig()`'s marker search both switch to recognizing either family (`OPENSPEC` or `RASEN`) for locating an existing block, and always construct new blocks with `RASEN` markers.
  - Uninstall (`removeProfileConfig()`, `removeBashrcConfig()`, zsh equivalent) must remove whichever family is actually present — a user who installed under the legacy marker and never re-ran configure still gets a clean uninstall.

**D3 — Marker literal chosen: `# RASEN:START` / `# RASEN:END` (shell-comment style), matching the existing `# OPENSPEC:START/END` syntax family used by these three installers.** The HTML-comment style `<!-- RASEN:START -->` referenced in `rasen-cli-identity/spec.md:59` is for markdown config files (a different code path, `OPENSPEC_MARKERS` in `config.ts`, already correctly described as recognize-only). This change only touches the shell-profile marker family; no change to `OPENSPEC_MARKERS`/`config.ts`.

**D4 — Delta placement for F6: ADDED scenario under `rasen-cli-identity`.** The requirement at `spec.md:59` already states installers "SHALL NOT" write the legacy marker into new content — the code fix is bringing existing code into conformance with an existing requirement, which by the project's conformance precedent needs no delta. However, the *dual-recognition* behavior (upgrade-replaces-legacy-block, uninstall-removes-either-family) is new-enough observable behavior that it deserves its own explicit scenario rather than being silently implied. This change adds one ADDED scenario to `rasen-cli-identity/spec.md` documenting: shell-completion installers write `RASEN` markers into new profile content; a pre-existing legacy `OPENSPEC` block is recognized and replaced in place on reconfigure; uninstall removes either marker family.

**D5 — A1/B1 remain plain conformance edits, no delta.** `isOpenSpecProject` → `isRasenProject` in `cli-completion/spec.md` is a token fix inside a fictional interface sketch (no such method exists either way — confirmed by reading `CompletionProvider`, which only has `getChangeIds()`/`getSpecIds()`). `config.yaml:9`'s "Write OpenSpec proposals" → "Write Rasen proposals" is a one-word prose edit inside a context bullet, not a structural YAML key — confirmed safe by direct read, nothing parses that sentence programmatically.

## Risks / Trade-offs

- [Risk] Flipping existing test assertions from `OPENSPEC` to `RASEN` on fresh-install paths could mask a real regression if the dual-recognition logic is subtly wrong (e.g., matching `RASEN` when it should only match on the exact `# RASEN:START` line). → Mitigation: keep the upgrade-path tests seeded with a literal legacy `OPENSPEC` block (not migrated to a new fixture format) so the replace-in-place behavior is exercised against a real legacy shape, and add an explicit "no duplicate block" assertion after upgrade.
- [Risk] `updateFileWithMarkers()` is a shared low-level utility (`src/utils/file-system.ts`); adding legacy-pair recognition could affect other undiscovered callers. → Mitigation: confirmed via grep that only `bash-installer.ts` and `zsh-installer.ts` call `updateFileWithMarkers()`; the new legacy-pair parameter is optional (defaults to no legacy recognition), so no other caller's behavior changes.
- [Risk] Rewriting 8 scenario blocks in `cli-update/spec.md` is a larger diff than a token swap and touches wording reviewers may parse closely. → Mitigation: keep each scenario's `WHEN`/non-marker `AND` clauses (file paths, frontmatter fields, argument placeholders) untouched; only the marker-mechanism `AND` clauses change.
- [Trade-off] Not consolidating the three duplicate marker-literal definitions leaves the duplication in place for a future cleanup. Accepted per Non-Goals — minimal fix, no scope creep.

## Migration Plan

No runtime migration. Existing installed shell profiles with a legacy `# OPENSPEC:START/END` block are handled by the upgrade path (D2): the next `rasen completion install`/reconfigure run replaces the block in place with `RASEN` markers; uninstall works against either marker family without requiring a prior reconfigure. No data migration, no version gate needed — this ships as a normal patch-level fix.
