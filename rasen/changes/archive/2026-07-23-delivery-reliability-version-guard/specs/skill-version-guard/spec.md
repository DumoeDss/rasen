## ADDED Requirements

### Requirement: Project-scoped commands warn when installed skills lag the running CLI

When a project-scoped command resolves its root and at least one configured tool's installed skills were generated by a CLI version different from the version currently running, the system SHALL print a single-line warning naming the installed version and the running version, plus a hint to run `rasen update`. This warning SHALL be suppressed entirely for `--json` output on every command that supports it, and SHALL never cause the invoked command to fail or exit non-zero.

#### Scenario: Mismatch warning on a non-update command

- **WHEN** a user runs `rasen show` (or any other project-scoped command) in a project whose installed skills were generated by CLI v0.1.2
- **AND** the running CLI is v0.1.5
- **THEN** the system SHALL print a warning naming both versions and suggesting `rasen update`
- **AND** the command's normal output and exit code SHALL be unaffected

#### Scenario: No warning when versions match

- **WHEN** a project-scoped command runs in a project whose installed skills were generated by the same version as the running CLI
- **THEN** the system SHALL NOT print the mismatch warning

#### Scenario: Warning suppressed under --json

- **WHEN** a project-scoped command that supports `--json` runs against a mismatched project with `--json` passed
- **THEN** the mismatch warning SHALL NOT appear in stdout, and the JSON payload SHALL be unaffected by the check

#### Scenario: Version-status lookup failure never breaks the command

- **WHEN** the underlying skill-version lookup throws or fails for any reason (e.g. an unreadable skill file)
- **THEN** the invoking command SHALL complete exactly as it would have without the check, with no warning printed and no error surfaced

### Requirement: Mismatch warning is debounced per version pair, not per invocation

The system SHALL avoid repeating the mismatch warning for the same (installed version, running version) pair within an ongoing sequence of commands against the same project, so a session issuing many project-scoped commands in a row (for example, an AI agent driving a multi-step workflow) is warned once rather than on every command. The warning SHALL fire again automatically once the pair changes — either because the installed version advanced (the project ran `rasen update`) or the running CLI's own version changed.

#### Scenario: Second command in a session is silent

- **WHEN** a project-scoped command triggers the mismatch warning for versions (0.1.2, 0.1.5)
- **AND** a second project-scoped command runs immediately after, in the same project, with no change in either version
- **THEN** the second command SHALL NOT repeat the warning

#### Scenario: Warning re-arms after the project updates

- **WHEN** the mismatch warning has already fired for (0.1.2, 0.1.5)
- **AND** the user runs `rasen update`, bringing the installed version to 0.1.5
- **AND** the running CLI is later upgraded again to 0.1.6 without another `rasen update` in the project
- **THEN** the next project-scoped command SHALL warn again, naming (0.1.5, 0.1.6)

#### Scenario: No debounce state available still warns

- **WHEN** a project has no machine-local registration to hold debounce state
- **THEN** the mismatch warning SHALL still print every time the condition is met, rather than being silently skipped

### Requirement: Doctor reports the skill/CLI version mismatch as a health finding

`rasen doctor` SHALL report a skill-version-mismatch finding, independent of the ambient warning's debounce state, whenever any configured tool's installed skills were generated by a version different from the running CLI. The finding SHALL appear in both human-readable and `--json` output and SHALL include a fix hint pointing to `rasen update`.

#### Scenario: Doctor reports mismatch in human output

- **WHEN** `rasen doctor` runs in a project with a skill/CLI version mismatch
- **THEN** the human-readable report SHALL include a note naming the installed and running versions
- **AND** SHALL include a `Fix: rasen update` line

#### Scenario: Doctor reports mismatch in JSON output

- **WHEN** `rasen doctor --json` runs in a project with a skill/CLI version mismatch
- **THEN** the JSON payload's status/finding list SHALL include an entry describing the mismatch, with the same information available in the human-readable form

#### Scenario: Doctor reports nothing when versions match

- **WHEN** `rasen doctor` runs in a project whose installed skills match the running CLI version
- **THEN** no skill-version-mismatch finding SHALL appear in either output form

#### Scenario: Doctor always re-checks regardless of the ambient warning's debounce state

- **WHEN** the ambient mismatch warning has already fired and been debounced for the current version pair in the same session
- **AND** the user then runs `rasen doctor`
- **THEN** doctor SHALL still report the finding, since doctor is an explicit, on-demand health check independent of the ambient warning's suppression
