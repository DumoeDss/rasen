## Why

A project's installed skills can silently fall behind the CLI that generated them. `rasen update` already stamps each installed skill file with the version it was generated by and compares that stamp to the running CLI's own version — but that comparison only ever runs *inside* `update` itself. Any other project-scoped command (`show`, `doctor`, `validate`, `work`, an agent-driven session running dozens of commands in a row) never looks at the stamp at all. In a same-day incident, a project's skills were generated by a CLI version that predated a same-morning global CLI refresh; the project never ran `update` again, and an AI-agent session went on to execute a stale, 3-day-old playbook for the rest of its run while believing it was current. The gap is not "no version is tracked" — it's "the tracked version is never surfaced outside `update`."

## What Changes

- Surface a one-line staleness warning ("skills were generated by CLI vX; the running CLI is vY — run `rasen update`") from the shared root-resolution path that ten project-scoped commands already funnel through (`doctor`, `show`, `validate`, `work`, `context`, `pipeline`, `pipeline-library`, `workflow instructions`, `workflow new-change`, `workflow status`), reusing the existing per-tool version-status check — no new stamping mechanism, no new detection logic.
- The warning is suppressed for machine-readable (`--json`) output, and is debounced per (stamp version, CLI version) pair so a long-running session (especially an AI agent driving many commands back-to-back) is warned once, not on every invocation; it re-arms automatically the moment the mismatch resolves or the CLI changes again.
- Add a `rasen doctor` health-report entry for the same mismatch, in both human and `--json` output, with a `Fix: rasen update` hint — so an explicit health check catches it even in a session that never trips the ambient warning.
- Document (spec-only, no behavior change) the existing `generatedBy` version-stamp write that `rasen init` and `rasen update` already perform on every installed skill file, since the new warning and doctor check now depend on it as load-bearing infrastructure and it was previously undocumented.

## Capabilities

### New Capabilities
- `skill-version-guard`: the mismatch-warning behavior at project-scoped command resolution (reusing existing version-status detection, gated off machine output, debounced), plus the `rasen doctor` check item for the same mismatch.

### Modified Capabilities
- `cli-init`: document the existing (unchanged) `generatedBy` CLI-version stamp written to each skill file at install time.
- `cli-update`: document the existing (unchanged) `generatedBy` CLI-version stamp written to each skill file at update time. Update's own already-correct "up to date (vX)" / "Updated: ... (vX)" output is untouched.

## Impact

- `src/core/root-selection.ts`: new call site in `resolveRootForCommand()`, alongside the existing `emitStoreRootBanner`/`touchProjectRegistry` calls, gated on `!json`, best-effort (never throws, never slows the command).
- `src/core/shared/tool-detection.ts`: reuse `getAllToolVersionStatus`/`getToolVersionStatus` (already exported, currently unused outside `update.ts`) — no changes expected here beyond possibly exporting a helper.
- New small module (or addition alongside `src/core/expert-selection-state.ts`) for the per-project debounce marker (last-warned version pair).
- `src/core/config-diagnostics.ts`: new `ConfigDiagnosticKey` (e.g. `skillVersionMismatch`) following the `expertSelectionMigration` pattern.
- `src/locales/{en,ja,zh-cn}.json`: new locale string under `configErrors.diagnostics` for the new key.
- `src/commands/doctor.ts` + `src/core/relationship-health.ts`: new health-report finding, human + `--json`.
- No template changes, no version bump (version is read-only from `package.json`), no new config-key-registry entry (no new user-settable key).
