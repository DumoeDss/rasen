## Why

Rasen ships every workflow as two nearly-identical prompt bodies — a skill (`.claude/skills/rasen-*/SKILL.md`) and a slash command (`.claude/commands/rasen/*.md`) — generated by parallel template functions per workflow. Claude Code has since merged custom commands into skills (a skill directory already produces the same slash command), and 15+ other tools natively discover project skills. The command delivery surface is now pure duplication: two bodies to keep in sync (they have already drifted), a 26-adapter generation module, and a `delivery` config dimension whose only remaining choice is "also emit the redundant copy." This change retires the entire command delivery surface so skills are the single source, per the approved design doc `rasen/office-hours/skills-only-delivery-runtime-next-steps.md` (Phase A).

## What Changes

- **BREAKING (config surface):** The `delivery` global-config setting is retired. Any stored `delivery` value — current (`both`/`skills`) or legacy (`commands`/`commands-first`/`skills-first`) — is read without error, triggers a one-time retirement notice, and is removed from the config on next write. The interactive delivery picker and `config set delivery` UX are removed.
- **Removal:** The entire `src/core/command-generation/` module (26 tool adapters + generator/registry/types) is deleted. Before deletion, the static knowledge needed for cleanup (the built-in command file IDs and each tool's command-path rules) is frozen into a small retained module, modeled on the existing `RETIRED_WORKFLOW_COMMAND_IDS` precedent in `legacy-cleanup.ts`.
- **Removal:** All 19 `get*CommandTemplate` template exports and the `CommandTemplate` type/`CommandContent` plumbing (`skill-generation.ts`, `templates/`, `workflow-registry` `command` field) are deleted. Skill template bodies are NOT touched in this change (that is a later phase).
- **Behavior change:** `rasen init` on a new project generates only skills and leaves no rasen command files. `rasen update` unconditionally cleans up any pre-existing rasen command files (merging the two current delivery-gated cleanup paths into one unconditional legacy cleanup) while never touching user-authored non-rasen files.
- **i18n:** New retirement/cleanup notices land in `en`/`ja`/`zh-cn` locales; retired delivery UX strings are removed.
- **Docs:** `docs/commands.md`, `docs/how-commands-work.md`, `docs/supported-tools.md`, `docs/customization.md`, `docs/getting-started.md`, and `docs/zh/` equivalents are updated; CHANGELOG entry added (version attribution deferred to the user; no version bump).
- **Non-goal:** No skills index is injected into any tool's instructions file (AGENTS.md / .cursorrules / GEMINI.md); natively-supported tools auto-discover skills.

## Capabilities

### New Capabilities

None. The cleanup behavior for retired command files is added as a requirement to the existing `legacy-cleanup` capability rather than a new capability, matching the `RETIRED_WORKFLOW_COMMAND_IDS` precedent.

### Modified Capabilities

- `command-generation`: Entire capability retired — all requirements REMOVED (the command generation subsystem no longer exists).
- `profiles`: The `delivery` dimension is retired. The "Delivery is independent of profile" and "Legacy delivery values migrate gracefully" requirements are rewritten from a two-value setting into a fully retired setting (any stored value → notice + removal); the interactive picker no longer offers a delivery choice.
- `cli-init`: Command-generation and command-count scenarios REMOVED; skills-only generation plus opportunistic legacy command cleanup ADDED.
- `cli-update`: The delivery-gated command-generation scenarios and the per-tool "Slash Command Updates" requirement are REMOVED; a single unconditional legacy command-file cleanup requirement is ADDED.
- `cli-config`: The `config profile` delivery prompt/flow and the `config set delivery` legacy-consolidation scenarios are REMOVED; a retired-key handling scenario (graceful notice, no persistence) is ADDED.
- `config-key-registry`: The `delivery` key is REMOVED from the settable-key registry; the registry-vs-schema drift test scenario is updated.
- `legacy-cleanup`: ADDED requirement — a static retired-command-paths cleanup that removes built-in rasen command files (current, `-command`-suffixed, and legacy `opsx`-prefixed variants) across all tool layouts without depending on the deleted live registry.
- `workflow-template-parity`: The command-template hash-map entries and the "workflow command templates pinned" scenario are REMOVED (command templates no longer exist); skill-template parity is unchanged.
- `workflow-library`: The workflow-definition `command` field/template registration is retired; the `ff`-specific "no command template" clause is generalized to the whole built-in set.
- `methodology-expert-fusion`: Scenarios that inspect "the generated skill and command templates" are narrowed to skill templates only (command templates are gone); expert body content is otherwise unchanged.

## Impact

- **Deleted:** `src/core/command-generation/` (whole directory, 26 adapters + generator/registry/types/toml/yaml/command-file-id).
- **Modified core:** `src/core/global-config.ts` (retire `Delivery` type, `DEFAULT_CONFIG.delivery`, `LEGACY_DELIVERY_MAP`, rewrite `normalizeDelivery` as retirement detection), `src/core/init.ts`, `src/core/update.ts` (delivery branches removed, cleanup merged), `src/core/shared/skill-generation.ts`, `src/core/templates/**` (19 CommandTemplate exports + types), `src/core/migration.ts`, `src/core/profile-sync-drift.ts`, `src/core/workflow-artifact-ledger.ts`, `src/core/workflow-registry/types.ts`, `src/commands/profile-editor.ts`.
- **New retained module:** `src/core/shared/retired-command-paths.ts` (frozen command IDs + per-tool path rules) and `src/core/shared/yaml.ts` (migrated `quoteYamlValue`).
- **UX/commands:** `src/commands/config.ts`, `src/commands/profile.ts`, profile/config editors; three locale files.
- **Tests:** command-generation tests deleted; init/update/config tests re-asserted (zero command files, unconditional cleanup, legacy-value notice, user-file safety); parity golden master updated.
- **Docs:** command-focused docs and their `zh/` mirrors; CHANGELOG.
- **Additional consumers surfaced by type deletion (beyond the doc's named `migration.ts` / `profile-sync-drift.ts`):** `workflow-artifact-ledger.ts`, `profile-editor.ts`, and the `command` field in `workflow-registry/types.ts` all import from `command-generation` and must be rewired to the static module or have the command concept dropped. `codex-home.ts` carries only a stale prose comment (no code dependency; used widely elsewhere).
- **No version bump.** No changes to pipeline registry or LEAD orchestration.
