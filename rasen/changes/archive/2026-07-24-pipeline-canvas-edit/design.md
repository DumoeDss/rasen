## Goals / Non-Goals

**Goals:**
- Full composition loop in the browser: assemble → live feedback → validate → save, with server guarantees untouched.
- Editing surface that cannot silently lose definition content (unexposed fields pass through verbatim).
- New-pipeline creation without hand-written YAML.

**Non-Goals:**
- No layout persistence (the YAML definition has no coordinates; positions are session-only, reopen re-layouts).
- No editing of built-ins (save-as from a built-in's definition IS possible via the new-draft flow seeded from its detail — read-only source untouched).
- No goal-loop gate authoring (measure/evaluate loop configs render read-only and are preserved verbatim; only review-cycle kind + maxRounds are editable).
- No minimap, no undo/redo stack (Discard is the escape hatch; undo/redo is future work if QA demands it).

## Decisions

1. **Same route, in-page mode toggle — not a separate edit route.** `/p|s/:id/pipelines/:name` keeps one lazy chunk, one detail fetch, one guard for `editable`; an Edit button (hidden when `editable: false`) flips to edit mode, Save/Discard flip back. Rationale: a `/edit` sibling route would duplicate fetch/not-found/error handling, complicate the dirty-navigation guard (route change = potential state loss), and create a deep-linkable URL whose meaning depends on unsaved local state. The new-draft case enters the same page with an initial-mode hint (see D6). Rejected alternative: `?mode=edit` query — same duplication issues without benefits, and preact-iso route params don't model it cleanly.

2. **Draft state lives in a pure module, `canvas/draft.ts`, beside `layout.ts`.** The draft is a `WirePipelineDefinition` value (NOT React Flow nodes) — the single source of truth; flow nodes/edges are derived per render via the existing `definitionToGraph`/`layoutGraph` on entry and incrementally maintained afterwards. Pure functions, each unit-testable: `addStage(def, {skill, role, id})`, `removeStage(def, id)` (drops the stage AND every `requires` reference to it), `addRequire(def, from, to)`, `removeRequire(def, from, to)`, `updateStageFields(def, id, patch)` (spread-patch — untouched fields preserved verbatim), `renameStage(def, oldId, newId)` (rewrites `requires` references), `wouldCreateCycle(def, from, to)` (reachability from `from` back to `to` over existing requires — the demo's algorithm), `stageIdFor(skill, def)` (skill-derived id + numeric uniquifier, panel-editable after). Keeping mutations off React Flow state means field preservation and cycle logic are testable without a canvas mount — same reasoning that made `layout.ts` pure.

3. **Editing surfaces: palette left, properties panel right (as recommended).** Palette: catalog skills (name + description tooltip; `enabled: false` greyed with "disabled in profile" title, still draggable? NO — greyed AND not draggable, because a dropped disabled skill guarantees a validation error; the tooltip explains why). HTML5 DnD onto the canvas creates a stage at `screenToFlowPosition` (demo-proven under compat). Properties panel opens on node selection: id (rename with reference rewrite), role/verifyPolicy/runtime dropdowns and gate checkbox from CATALOG vocabularies (never literals), skill picker (enabled skills), condition (freeform input + catalog's suggested labels as datalist), parallelGroup (freeform + existing groups datalist), model (KNOWN_MODEL_IDS datalist like existing pages), loop: kind selector limited to none/review-cycle with maxRounds number; a stage carrying a goal-loop config shows a read-only JSON summary with "preserved as-is" note. Pipeline name (locked once saved — rename is out of scope; name is the save identity) and description edit in the header.

4. **Group/drag interaction under the `parentId` contract.** Free stages drag with plain position updates. Group containers themselves stay non-draggable (as child 3 built them); members drag within `extent: 'parent'`. Changing a stage's `parallelGroup` in the panel is a STRUCTURAL edit: the draft updates, then groups are recomputed via a full `layoutGraph` re-run (accepting a layout jump — cheaper and more predictable than incremental group-box math, and preserves the group-before-member ordering invariant). A "Re-layout" toolbar button re-runs auto-layout on demand.

5. **Validation overlay maps issues by locator path.** `POST /api/v1/pipeline-validation` on the Validate button AND automatically before save. Mapping in pure `issuePathTarget(path)`: `/stages/<i>/...` → the stage id at index i in the DRAFT's stage order (nodes get an error/warning ring + badge; the panel field named by the tail segment highlights when that stage is open); `/stages` or `/` → pipeline-level, shown only in the issues drawer; unmappable paths degrade to the drawer (never dropped). The drawer lists every issue with severity, message, and a click-to-select-the-node affordance. Client-side `wouldCreateCycle` gives instant connect-time rejection (transient toast, demo pattern); the server dry-run remains authoritative — the overlay renders SERVER issues, the client check is only a fast-path UX guard.

6. **New-draft flow: name-first dialog, no reserved URL.** "Assemble in canvas" on the Pipelines page asks for a name (validated against the id grammar client-side), then navigates to the graph route for that name with a navigation-state hint (in-memory, e.g. a module-level pending-draft record consumed on mount) that mounts the page in edit mode with a minimal draft `{ name, origin: 'ui', stages: [] }` instead of fetching (a fetch would 404). Because the hint is in-memory, a hard refresh of an unsaved draft's URL degrades to the 404 not-found view — acceptable and honest (nothing exists server-side yet); the not-found view gains a "Start assembling '<name>'" affordance to recover. Save-as for built-ins: the read-only view of a built-in gets "Duplicate to edit" which runs the same flow seeded from the built-in's definition (name prompted, origin stamped 'ui').

7. **Save flow and its failure UX.** Save button: (1) dry-run validate — error-severity issues open the drawer and abort (the floor violation for a UI draft therefore always surfaces here first); (2) `mutatePipeline({ op: 'save', name, definition: draftWithOriginUi, force })` — `origin: 'ui'` stamped at this boundary (idempotent if already present, e.g. duplicated drafts); (3) 201/200 → clear dirty, refetch detail, exit to view mode; 422 with an existing-name refusal and no prior `force` → offer explicit "Overwrite" retry (the import-dialog precedent); 422 otherwise (built-in name, floor race) → verbatim CLI message in the drawer area; 409 busy → "another library operation is running, retry" message, no auto-retry loop.

8. **Dirty-state protection, three layers.** (a) visible "Unsaved changes" chip in the header once the draft diverges (deep-equal against the loaded definition); (b) in-app: Back link and mode-exit ask for confirmation while dirty (Discard/Keep editing); (c) `beforeunload` handler registered only while dirty (removed on save/discard/unmount) for tab close and hard navigation. No draft persistence to localStorage — out of scope, listed as future work rather than half-shipped.

9. **Mirror + client additions (first-consumer discipline).** `types.ts`: `PipelineValidationRequest/Issue/Response`, `PipelineCatalogSkill/Response`, `save` member appended to `PipelineMutationRequest`; drive-by tightening `PipelineAgentRuntimeConfig.runtime` to required (matches core parsed-output type — review-accepted Minor from child 3). `client.ts`: `validatePipeline(definition, selector)`, `getPipelineCatalog()`. Catalog fetched once per editor entry and cached for the page's lifetime (vocabulary is installation-wide and changes only with installs).

## Risks / Trade-offs

- [Silent data loss through the editor (R4's UI twin)] → the draft IS a full `WirePipelineDefinition`; mutations are spread-patches; a dedicated test feeds a definition using EVERY loader field (agents, handoff, reuse, goal-loop configs, sessionReuse/sandbox/effort), edits one unrelated field, and asserts byte-level preservation of the rest in the posted save body.
- [Issue-path index mapping drifts from draft stage order] → indices resolve against the same draft array the validation request serialized; `issuePathTarget` is pure and tested; unmappable paths degrade to the drawer, never dropped.
- [In-memory new-draft hint lost on refresh] → accepted (D6) with an explicit recovery affordance on the not-found view; alternative (draft in sessionStorage) deferred with undo/redo as future work.
- [Client cycle check disagrees with server (e.g. decompose constraints)] → client check only BLOCKS a plainly cyclic connect; everything else is allowed locally and caught by the authoritative dry-run — the editor never claims validity itself, the Save gate does.
- [React Flow interaction breakage under preact/compat beyond the demo's surface] → the demo exercised exactly this surface (DnD palette, `screenToFlowPosition`, connect handlers, state hooks); fallback remains the documented React island; browser-level verification via rasen-qa is part of this child's verification tasks.
- [Group re-layout jump on parallelGroup edits disorients] → accepted trade (D4): predictable auto-layout over incremental box math; the Re-layout button makes layout ownership explicit to the user.
