## Why

The verify → ship → archive lifecycle has three broken seams. (1) `ship`'s pre-flight tells the user to "Run /rasen:verify first," but the plain `/rasen:verify` (verify-change) writes no report file at all — ship never finds evidence, warns forever, and the user learns to routinely bypass the verification gate (WF-1). (2) The two verify entry points speak different verdict vocabularies (verify-change: CRITICAL/WARNING/SUGGESTION + "Ready for archive"; verify-enhanced: per-stage PASS/FAIL + "must fix before shipping"), and neither maps to a machine-checkable status any downstream stage can gate on (WF-7). (3) ship's evidence-based test-skip needs a recorded test run plus a content-tree fingerprint, but only the review-cycle loop records one — the standard verify path produces reports with no fingerprinted evidence, so the documented skip optimization is unreachable through it (WF-8). Separately, the shared `QA_METHODOLOGY` and `DESIGN_METHODOLOGY` blocks state "Never read source code" as an un-scoped absolute under "Important Rules," which directly contradicts their own diff-aware triage ("map changed files to the routes/pages they serve") and their standalone fix loops ("Read the source code, make the minimal fix") — on the common `/qa` and `/design-review` feature-branch path (SH-1, SH-2, RV-6, RV-7).

## What Changes

- **Verify writes a canonical evidence file (WF-1).** `verify-change` persists a `verification-report.md` to the change directory (summary scorecard + verdict + findings). `ship`'s pre-flight accepts it as verification evidence alongside the expert `*-report.md` and `review-cycle-report.md` files (the canonical reports child #1 established), so there are no orphan consumers and the "Run /rasen:verify first" loop is closed.
- **One verdict vocabulary + machine-checkable status line (WF-7).** Both verify variants map their findings to the canonical Blocker/Major/Minor/Trivial scale (consuming child #1's `canonical-severity-vocabulary` capability — not re-declaring it) and emit a shared, parseable status line into their written report. A verify "pass" means no open Blocker or Major, matching the review-cycle termination invariant. (Whether incomplete tasks hard-block archive is the WF-4 archive-gate contract — child #5 — and is out of scope here; this change unifies vocabulary and the pass rule only.)
- **Close the test-evidence chain (WF-8).** When a verify variant runs the project test/gate suite as part of verification, it records the exact command(s), the result, and `git rev-parse HEAD^{tree}` into its report — the same fingerprinted-evidence schema `review-cycle-report.md` already uses. `ship`'s existing skip gate (which already reads "another verification report") then honors the skip. ship's evidence list names `verification-report.md` explicitly.
- **Scope "Never read source code" to its real intent (SH-1, SH-2, RV-6, RV-7).** Using the enumerate-and-gate idiom, add a scope clause to the `QA_METHODOLOGY` and `DESIGN_METHODOLOGY` "Important Rules" that names every absolute swept from the block and carves out the two legitimate source-reading activities: (a) diff-aware triage mapping changed files → affected routes/pages, and (b) the standalone fix loop. The rule keeps its real intent: do not read source to FORM findings during exploration/testing.
- **Add chrome-use to the parity golden master.** `chrome-use.ts` embeds the PREAMBLE but is absent from the parity hash table, so PREAMBLE changes ship unverified for it (accepted-known debt surfaced by child #1). Add its function-hash and generated-content-hash entries so future shared-block changes are caught.
