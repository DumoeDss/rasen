## ADDED Requirements

### Requirement: Generated workflows receive capability-matched orchestration bundles

Rasen SHALL generate each orchestration entry workflow with one self-contained playbook bundle selected by that workflow's executable capabilities.

#### Scenario: Auto receives the complete orchestration bundle

- **WHEN** the `rasen-auto` skill is generated
- **THEN** its playbook SHALL include the shared header, capability tier (`A` and `A.1`), dispatch (`B`, `B.2`, `B.3`, and `B.4`), author-verifier (`C`), stage metadata (`D`), review loop (`E`), goal loop (`L`), run-state (`F` and `F.1`), portfolio (`G` and `G.1`), and context-handoff (`H`) modules
- **AND** it SHALL include the persistent-planner module (`B.1`)

#### Scenario: Goal receives only goal-applicable orchestration modules

- **WHEN** the `rasen-goal` skill is generated
- **THEN** its playbook SHALL include the shared header, capability tier (`A` and `A.1`), dispatch (`B`, `B.2`, `B.3`, and `B.4`), author-verifier (`C`), stage metadata (`D`), goal loop (`L`), run-state (`F` and `F.1`), and context-handoff (`H`) modules
- **AND** it SHALL omit persistent planner (`B.1`), review loop (`E`), and portfolio (`G` and `G.1`) modules

#### Scenario: Review cycle receives only review-applicable orchestration modules

- **WHEN** the `rasen-review-cycle` skill is generated
- **THEN** its playbook SHALL include the shared header, capability tier (`A` and `A.1`), dispatch (`B`, `B.2`, `B.3`, and `B.4`), author-verifier (`C`), review loop (`E`), run-state (`F` and `F.1`), and context-handoff (`H`) modules
- **AND** it SHALL omit persistent planner (`B.1`), stage metadata (`D`), goal loop (`L`), and portfolio (`G` and `G.1`) modules

### Requirement: Selective bundles remain complete and behavior-preserving

Each generated bundle SHALL preserve the original relative order and semantics of every module it includes, and SHALL contain no instruction that directs its lead to a module omitted from that bundle.

#### Scenario: Included modules retain canonical ordering

- **WHEN** any named orchestration bundle is composed
- **THEN** its included module headings SHALL appear in the same relative order as the complete auto playbook
- **AND** every rule shared by multiple bundles SHALL come from one canonical source definition

#### Scenario: Narrow bundles have no dangling optional-step references

- **WHEN** the generated goal or review-cycle playbook is inspected
- **THEN** references to excluded step identifiers SHALL be absent
- **AND** core dispatch, author-verifier, run-state, resume, and handoff rules SHALL remain present

#### Scenario: Existing runtime contracts remain unchanged

- **WHEN** the complete auto bundle and either narrow bundle execute a capability they share
- **THEN** capability detection, role-isolated dispatch, author-verifier enforcement, run-state, resume, context sensing, and handoff behavior SHALL match the existing orchestration contract

### Requirement: Selective inline remains self-contained

Each generated workflow skill SHALL carry its selected playbook inline and SHALL be usable without loading an orchestration sidecar.

#### Scenario: Installed skill is self-contained

- **WHEN** `rasen init` or `rasen update` generates any of the three orchestration entry skills
- **THEN** the resulting `SKILL.md` SHALL contain its selected playbook text directly
- **AND** invoking the skill SHALL require no additional orchestration source file

### Requirement: Generated bundle sizes stay within conservative ceilings

The UTF-8 byte size of each fully generated orchestration entry `SKILL.md` SHALL stay within a workflow-specific regression ceiling.

#### Scenario: Generated sizes satisfy budgets

- **WHEN** the auto, goal, and review-cycle skill files are generated by the template pipeline
- **THEN** `rasen-auto/SKILL.md` SHALL be no larger than 106 KB
- **AND** `rasen-goal/SKILL.md` SHALL be no larger than 70 KB
- **AND** `rasen-review-cycle/SKILL.md` SHALL be no larger than 60 KB

### Requirement: Template parity tracks selective-bundle output

The workflow parity suite SHALL pin the function payload and generated skill content of every orchestration entry template affected by bundle selection.

#### Scenario: Expected parity entries are updated

- **WHEN** selective bundle composition changes generated workflow text
- **THEN** the parity golden master SHALL contain current hashes for `rasen-auto`, `rasen-goal`, and `rasen-review-cycle`
- **AND** unrelated template hashes SHALL remain unchanged
