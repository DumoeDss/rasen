# Planning context: delivery-reliability portfolio (LEAD seed)

User intent (verbatim): 修复两个已拍板的产品缺口:① skill 版本失配告警;② orchestration playbook 四条派发协议修补。全角色 sonnet,decompose 为两个独立 child,portfolio 单次交付。

## Child 1: delivery-reliability-version-guard

**Incident evidence (2026-07-23, elftia project, session a657e84c):** user ran `rasen update` in the project BEFORE the global CLI was refreshed that morning. `rasen update` is write-if-changed (verified: second run prints "Use --force to refresh files anyway", mtime untouched) and prints nothing about "0 files changed" or which template version it delivered. Project skills stayed at their 7/20 install; the session's LEAD ran a 3-day-stale playbook; the user believed they were on latest. Silent + no version visibility = the gap.

**Fix direction (planner refines):**
1. Stamp installed skills with the CLI version at install/update time (e.g. a single meta file under the tool's skill root, NOT per-skill frontmatter churn; read version from package.json — never bump/modify versions, 版本号归用户管).
2. Mismatch detection: when the CLI runs in a project whose stamp ≠ current CLI version, print a one-line warning + `rasen update` hint. Choose the detection point(s) deliberately — cheap check, no per-command overhead explosion; candidates: commands that already resolve a project root; consider debounce (warn once per day/session?) to avoid nagging.
3. `rasen update` output: always state what happened — "updated N files / all unchanged (templates vX)" — kill the silence.
4. `rasen doctor`: add a stamp-vs-CLI check item.
5. New user-facing strings go through the locale files (en/ja/zh-cn); config keys (if any) must be registered per config-key-registry spec.

**Touch surface guess:** src/core (update/init install path, doctor), locales ×3, tests. NO template changes.

## Child 2: delivery-reliability-playbook

Four dispatch-protocol fixes to the shared orchestration playbook, each with same-day incident evidence:

1. **Worker return must be delivered via SendMessage.** Evidence: harness background-agent model drops a worker's final plain text; impl-m3's apply DONE was lost (LEAD reconstructed from blackboard); after adding an explicit "return via SendMessage to main" clause to dispatch prompts, all subsequent workers delivered reliably. Fix: bake the clause into Step B's dispatch template and H.3's return contract (DONE and HANDOFF both).
2. **Post-handoff stale-instruction immunity.** Evidence: impl-zh sent HANDOFF, then a stale earlier correction message in its inbox re-woke it; it resumed editing (reverted its own edits) colliding with the already-dispatched successor; user had to interrupt manually. Fix: stand-down/handoff protocol gains "after returning HANDOFF/DONE-standDown, treat any inbound instruction issued BEFORE your handoff as expired — acknowledge and remain idle"; LEAD side: after accepting a HANDOFF, do not send further work to the predecessor.
3. **Apply-implementer review-window parking seam.** Evidence: elftia session — implementer (254k context) dispatched ONE_SHOT for apply, idled 8.8min during review, TTL expired, paid 226k rewrite when fixes routed back. B.4's LOOP_BOUND covers review-loop reviewer/fixer but NOT the apply implementer awaiting its first review verdict. Fix: extend the reuse-horizon guidance — when a verify/review-loop stage follows apply and the implementer's context ≥ the keepalive floor, dispatch it with a horizon that parks it (in `rasen agent wait --role impl…`) pending the first review verdict; stand it down when review is clean or fixes are re-routed.
4. **Message batching discipline.** Evidence: elftia archiver got 2 SendMessages 20s apart — each rebased its conversation (53k+66k double tax). Fix: LEAD-side rule in Step B/E: consecutive instructions to the same non-parked worker are batched into ONE message whenever no intermediate result is needed.

**Touch surface guess:** src/core/templates/workflows/_orchestration.ts (+ review-cycle.ts if it mirrors the wording), skill-templates-parity hash update, possibly docs. NO core-logic changes.

## Unit 1 findings (delivery-reliability-version-guard, proposed)

- The seed's fix-direction point 1 ("stamp installed skills with CLI version") was ALREADY IMPLEMENTED before this change: `generateSkillContent()` (src/core/shared/skill-generation.ts) writes a `generatedBy: "<version>"` field into every skill file's frontmatter, from both `rasen init` (src/core/init.ts) and `rasen update` (src/core/update.ts), always sourced read-only from package.json. Detection is also already implemented and cheap: `getToolVersionStatus`/`getAllToolVersionStatus` (src/core/shared/tool-detection.ts) reads only the first existing skill file per tool. `getAllToolVersionStatus` was exported but UNUSED outside update.ts — a ready-made hook.
- The real gap (confirmed against the incident timeline) is narrower than the seed guessed: mismatch detection only ever ran inside `update` itself. Nothing warned when a session ran OTHER project-scoped commands after the global CLI was refreshed but before the project re-ran `update`.
- `rasen update`'s own output already states version info correctly ("✓ All N tool(s) up to date (v0.1.5)", "✓ Updated: ... (v0.1.5)", plan preview shows "fromVersion → toVersion") — seed's point 3 needed NO change; left cli-update's existing update-output scenarios untouched.
- Chose `resolveRootForCommand()` (src/core/root-selection.ts) as the single hook point — 10 commands already funnel through it (doctor/show/validate/work/context/pipeline/pipeline-library/workflow instructions/new-change/status), and it already has the exact precedent shape needed: `emitStoreRootBanner(...)` gated on `!json`, and `touchProjectRegistry(...)` documented as "best-effort, throttled... must never fail or visibly slow a command" — reused both precedents directly.
- Debounce design: NOT time-based (rejected the `touchProjectRegistry` 24h-TTL pattern) — keyed on the (stampVersion, cliVersion) pair instead, because the incident's actual victim was an AI-agent session running many project-scoped commands back-to-back, where a TTL would either nag through the whole session or still fire multiple times. Marker storage reuses the `expert-selection-state.ts` precedent exactly (`resolveProjectHome(root.path, { ensure: false })` + a small JSON marker file in the project's machine-local home; best-effort, silently skipped if no home is registered — falls back to warning every time rather than forcing home registration as a side effect of a warning feature).
- `rasen doctor` gets an UNDEBOUNCED check for the same mismatch (own health finding, human + `--json`, `Fix: rasen update`) — doctor's contract is "explicit on-demand health report," so it always re-derives from `getAllToolVersionStatus` rather than consulting the ambient warning's debounce state.
- No dedicated "doctor" capability spec exists in rasen/specs/ — doctor's check items are documented piecemeal inside whichever capability spec owns each finding. Put the new doctor scenario inside the new `skill-version-guard` capability spec rather than hunting for a nonexistent central doctor spec.
- New diagnostic key `skillVersionMismatch` added to `CONFIG_DIAGNOSTIC_KEYS` (src/core/config-diagnostics.ts), following the existing `expertSelectionMigration` pattern exactly (same `reportConfigDiagnostic` shape); locale strings added to `configErrors.diagnostics` in all three locale files (en/ja/zh-cn, confirmed same line ~559 parent-path structure in all three today).
- cli-init and cli-update specs previously did NOT document the `generatedBy` stamp write at all (pre-existing spec gap). Closed it with ADDED (not MODIFIED) requirements in each delta spec — pure documentation of existing, unchanged behavior; zero code change in those two files.
- Deliberately did NOT add any new user-settable config key / opt-out flag — kept config-key-registry untouched; not requested by the fix direction.
- Change created and validated: `rasen validate delivery-reliability-version-guard --strict` → "Change 'delivery-reliability-version-guard' is valid". Artifacts: proposal.md, design.md, tasks.md, specs/{skill-version-guard (ADD), cli-init (ADD scenario), cli-update (ADD scenario)}/spec.md, all under rasen/changes/delivery-reliability-version-guard/.
