## Findings

### Major

**M1. Store-mode + in-ship timing loses the PR-body delta-spec embedding — the exact material the mechanism exists to carry.** `src/core/templates/workflows/ship.ts:84,119,131-132`. The store-mode embedding reads delta specs fresh from `<changeRoot>/specs/**/spec.md` at PR Body Generation time, but under in-ship timing the change directory (including `specs/`) moved or was deleted in step 3(b). Step (b).1's capture list names only proposal PR-body sections and task-completion facts — not delta specs — and the prose gives no pointer to the archived path for specs (it gives exactly that warning-and-remedy for the proposal). Failure scenario: store-rooted change, `archive.timing: in-ship`, mode `pr` → step (b) moves the change dir into the store's archive → embedding step finds `<changeRoot>/specs/**` gone → PR ships without its review material or with a false "absent" claim. Fix shape: add delta-spec content (or the moved path recorded in step (b).3/(b).4) to the (b).1 capture list when `root.store_id` is present.

**M2. Store-mode + in-ship: the new `## Archive` section records a false "fact".** `src/core/templates/workflows/ship.ts:180-182` asserts `Archive commit: identical to Commit: above — in-ship bookkeeping and delivery share one commit`. For a store-rooted change the in-ship bookkeeping (spec sync into store-side main specs + change-dir move within the store) mutates the STORE repo; ship's own commit (step (b)) is a code-repo commit and cannot contain it. Nothing in the flow commits the store repo, so: (a) the store is left with uncommitted sync/move mutations, and (b) the recorded chain claims an archive commit that contains no bookkeeping — violating this change's own recorded-facts honesty discipline (D1/D2 rationale; cf. the D3 dirty-tree honesty rule). Root cause (store-blind in-ship bookkeeping, `ship.ts:83-93`) predates this child; the new prose upgrades the gap into an affirmatively wrong recorded fact. Needs a LEAD decision: qualify the in-ship `## Archive` prose for store mode (e.g. record the store-side state honestly, mirroring D3's dirty-tree stamp), or record store×in-ship as a known-open follow-up alongside the existing store write-back follow-up.

### Minor

Mitigation: the authoritative step-8 body (lines 135/413) is byte-identical in both getters and carries the full consent-separation rule; the drift is in the redundant restatement only — the same class the author's own lesson-learned paragraph warns about.

**M4. The prescribed archive commit message asserts "(specs synced)" unconditionally.** `archive-change.ts:168-170` (both getters), bulk 8b equivalents. Step 4 explicitly permits "Archive without syncing," and a change may have no delta specs at all; the only message forms offered both claim "specs synced". Before this change there was no `-m` template, so the agent worded the commit accurately. The new no-ship fallback form exactly matches the repo's four prior archive commits (`chore(rasen): archive <name> (specs synced)`) — convention conformance is otherwise good. Suggest one parenthetical permitting an accurate variant when sync was skipped / no delta specs existed.

**M5. Ship-log `Store:` line names the wrong source fields.** `ship.ts:164`: "the registered store's path, from `root.store_id`/`changeRoot`" — the store's path is `root.path` (which the `git -C` stamps at `ship.ts:134-135` correctly use); `store_id` is an identity string and `changeRoot` is the change's directory, not the store root. An agent following the annotation literally stamps the wrong value. One-line prose fix.

**M6. Archive step 5.5's workDir fallback is incoherent for sticky-legacy logs under destructive destinations, and diverges from the prune tombstone's resolution.** `archive-change.ts:176` (both getters). (a) When the ship-log lives in `changeRoot` (legacy/sticky fallback), under `external` the file moves with the directory and under `prune` it is deleted; "resolve workDir before the bookkeeping" captures a path, not the file — appending to the old path resurrects `rasen/changes/<name>/` after the commit that removed it. `ship.ts:152` solves this identical problem by redirecting the fallback to the archived path; 5.5 lacks that instruction. (b) Prune's tombstone writer (step 5, `archive-change.ts:153`) MINTS a workDir when status carries none, while 5.5 says fall back to `changeRoot` — two writers of the same file within one run can target different locations, undermining the "tombstone … in the same ship-log" coexistence claim. Narrow (legacy log × destructive destination), but the fix is copying ship.ts's existing pattern.

### Trivial / observations

**T1. Interrupted archive half-closes the chain permanently.** If the run dies between the `## Archive` append and the post-commit `**Archive commit:**` journal line, a re-run no-ops at step 1.5 (already-archived) and the SHA line is never written. Acceptable journal semantics ("a journal, not a form"); future chain-record consumers must tolerate a missing `Archive commit:` line.

**T2. Out-of-scope, pre-existing store-safety gaps adjacent to the fixed one, same file:** `ship.ts:51` (pre-flight tasks read: literal `rasen/changes/<name>/tasks.md`) and `ship.ts:152` (ship-log fallback: literal `rasen/changes/<name>/ship-log.md`) misresolve for store-rooted changes — same class as the proposal-read bug this change fixed. For LEAD triage as follow-up; not fixed here per report-only mode.

**T3. Chain records are agent-workflow-only.** The CLI-side archive path (`src/core/archive.ts`, which writes prune tombstones) writes no chain records. Consistent with the proposal's templates-only scope; noting so the `sha-cross-stamping` capability isn't read as covering CLI-driven archives.

**T4. Store stamps are pr-mode-only by design reading.** `Store:`/`Store commit:` ship-log lines are scoped "pr mode, store-rooted change only" (`ship.ts:164-165`); a push/local store-rooted ship records no store SHA. This conforms to the delta specs' scenarios as written (embedding is PR-body-scoped), but leaves a chain hole in non-pr store deliveries — flag if chain completeness there ever matters.
