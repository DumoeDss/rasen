## Context

The gstack expert skills were integrated into OpenSpec via path replacement. The skill instructions (generated by `gen-skill-docs.ts` from `.tmpl` templates) still contain gstack-specific runtime infrastructure. The `generatePreamble` composite function in gen-skill-docs.ts generates a `{{PREAMBLE}}` block used by 23 of 28 skill templates. This block calls 9 sub-generators, most of which reference gstack binaries and branding.

Current preamble composition:
1. `generatePreambleBash` — 15 lines of bash (update-check, sessions, config reads, repo-mode)
2. `generateUpgradeCheck` — gstack version upgrade flow
3. `generateLakeIntro` — Boil the Lake principle intro with garryslist.org
4. `generateAskUserFormat` — AskUserQuestion format guidelines
5. `generateCompletenessSection` — Completeness principle with CC+gstack branding
6. `generateRepoModeSection` — Solo/collaborative repo ownership
7. `generateSearchBeforeBuildingSection` — Knowledge layers + search-first approach
8. `generateContributorMode` — gstack team field reports
9. `generateCompletionStatus` — DONE/BLOCKED escalation protocol

OpenSpec global config (`GlobalConfig` interface in `src/core/global-config.ts`):
```typescript
interface GlobalConfig {
  featureFlags?: Record<string, boolean>;
  profile?: Profile;
  delivery?: Delivery;
  workflows?: string[];
}
```

## Goals / Non-Goals

**Goals:**
- Remove gstack-specific features (upgrade, sessions, contributor mode, lake intro) from skill templates
- Migrate `proactive` and `repoMode` to OpenSpec global config
- Replace gstack branding with OpenSpec equivalents
- Produce a clean preamble that works without gstack binaries
- Remove gstack-upgrade skill entirely

**Non-Goals:**
- Porting review dashboard (gstack-review-read/log) — deferred
- Porting gstack-diff-scope — deferred
- Decoupling browse/ from gen-skill-docs.ts — deferred
- Renaming gstack binary names in deferred features

## Decisions

### D1: New Preamble — OpenSpec-native

Replace the 15-line gstack preamble bash with a minimal OpenSpec preamble:

```bash
## Preamble (run first)

\`\`\`bash
_BRANCH=$(git branch --show-current 2>/dev/null || echo "unknown")
echo "BRANCH: $_BRANCH"
\`\`\`
```

The `proactive` and `repoMode` values are read from OpenSpec global config at build time (by `openspec init`) and embedded directly in the skill instructions, rather than reading config at runtime via bash. This eliminates all gstack binary dependencies in the preamble.

**Rationale**: OpenSpec already knows user preferences at `openspec init` time. No need for runtime binary calls. If the user changes config, they re-run `openspec init --force`.

### D2: Config-Embedded Approach for Proactive and RepoMode

Instead of runtime `gstack-config get proactive`, embed the values in skill instructions during `openspec init`:
- Add `proactive?: boolean` and `repoMode?: 'solo' | 'collaborative'` to `GlobalConfig`
- In `generateSkillContent()` or a new `transformInstructions` callback, replace markers like `{{PROACTIVE_VALUE}}` and `{{REPO_MODE_VALUE}}` with actual config values
- Default: `proactive = true`, `repoMode = 'collaborative'`

**Alternative considered**: Keep runtime config reads via a simple `cat ~/.openspec/config.json | jq .proactive`. Rejected because it adds runtime file dependency and `jq` requirement.

### D3: Sections to REMOVE from generatePreamble

| Sub-generator | Action | Reason |
|---|---|---|
| `generatePreambleBash` | **REWRITE** | Replace with minimal branch-only bash |
| `generateUpgradeCheck` | **REMOVE** | OpenSpec has `openspec init --force` |
| `generateLakeIntro` | **REMOVE** | gstack branding, garryslist.org links |
| `generateAskUserFormat` | **KEEP** | Useful format guidelines, no gstack dependency |
| `generateCompletenessSection` | **REWRITE** | Keep philosophy, remove "CC+gstack" branding |
| `generateRepoModeSection` | **KEEP** | Useful feature, embed config value |
| `generateSearchBeforeBuildingSection` | **KEEP** | Useful approach, minimal gstack references |
| `generateContributorMode` | **REMOVE** | gstack-specific feedback mechanism |
| `generateCompletionStatus` | **KEEP** | Useful escalation protocol |

### D4: gstack-slug Replacement

Replace `source <(~/.openspec/bin/gstack-slug 2>/dev/null)` calls with inline bash:
```bash
SLUG=$(basename "$(git remote get-url origin 2>/dev/null)" .git 2>/dev/null || basename "$(pwd)")
```

This eliminates the gstack-slug binary dependency. Used in ~12 skills for project-scoped output paths.

### D5: Review Dashboard References — Soft Removal

For `gstack-review-read` and `gstack-review-log` calls (deferred feature), replace with comments:
```bash
# Review dashboard: deferred — see OpenSpec roadmap
```

Skills that display a "Review Readiness Dashboard" will skip that section. The dashboard concept can be re-implemented later as an OpenSpec-native feature.

### D6: gstack-upgrade Skill Removal

Delete `src/core/templates/experts/gstack-upgrade.ts`, remove from `experts/index.ts` barrel export, remove from `skill-generation.ts` registration. The `openspec-gstack-upgrade` directory will no longer be generated.

### D7: Branding Replacements

| Old | New |
|-----|-----|
| `CC+gstack` | `AI-assisted` |
| `gstack skills` | `expert skills` |
| `gstack follows the **Boil the Lake** principle` | OpenSpec's **Completeness Principle**: always do the complete thing when AI makes the marginal cost near-zero |
| `garryslist.org/posts/boil-the-ocean` | Remove entirely |
| `Hey gstack team` | Remove (contributor mode removal) |
| `gstack browse` | `browse` |

## Risks / Trade-offs

**[Review dashboard disabled]** → Skills referencing `gstack-review-read` will skip dashboard display. This is a degradation but the dashboard never worked without gstack binaries anyway.

**[Config not runtime-updateable]** → Embedding proactive/repoMode at init time means users must re-run `openspec init --force` to change them. Acceptable since these are infrequent preference changes.

**[Deferred binary references remain]** → `gstack-review-log`, `gstack-diff-scope` calls will be commented out but still visible in templates. This is intentional — they mark where future OpenSpec-native features can be wired in.
