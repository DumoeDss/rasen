# 编写好的规格

你很少会从一张白纸开始写规格。你用自然语言描述一个变更，`/opsx:propose` 起草需求和场景，然后你把它们改好。本页讲的就是最后这一步——什么是"好"，以及如何引导 AI 走向它。

它是[审查变更](reviewing-changes.md)的姊妹篇：审查是发现草稿中的薄弱之处，编写是知道一份强力的草稿由什么构成。

## 规格是行为，而非代码

规格描述的是你的系统*做什么*，用任何人都能核对的措辞来表达——而不是它如何构建。它由**需求**（行为的陈述）和**场景**（证明这些需求的具体示例）构成。

```markdown
### Requirement: Session Timeout
The system SHALL expire a session after 30 minutes of inactivity.

#### Scenario: Idle timeout
- GIVEN an authenticated session
- WHEN 30 minutes pass with no activity
- THEN the session is invalidated and the user must re-authenticate
```

把"怎么做"——队列、库、表结构——留在 `design.md` 或代码里。当行为和实现混进同一条需求时，这条需求就不再可测试，并且一旦代码变动就会开始过时。

## 什么构成一条好需求

一条好需求只表达一个行为，陈述得清楚到你可以直接交给别人去测试。

- **一条陈述，一个 `SHALL`/`MUST`。** 如果一条需求有三个"以及"，那它其实是三条需求。拆开它们。
- **可观察。** 代码之外的人应该能判断它是否成立。"系统 SHALL 在上传超过 10 MB 时显示错误横幅"是可观察的。"系统 SHALL 优雅地处理大文件上传"则不是。
- **强度恰当。** OpenSpec 使用 RFC 2119 关键字，它们含义不同：

  | 关键字 | 含义 |
  |---------|---------|
  | `MUST` / `SHALL` | 硬性要求。不可协商。 |
  | `SHOULD` | 强烈推荐，但允许有正当理由的例外。 |
  | `MAY` | 真正可选。 |

  默认使用 `MUST`/`SHALL`。只有当你确实想表达"除非有充分的理由不这样做"时，才使用 `SHOULD`。

判断一条需求的标准：*一个从未看过代码的测试人员能否判断它是否通过？* 如果不能，就需要打磨。

## 什么构成一个好场景

场景是需求证明自身价值的地方。每一个都是一条具体的 GIVEN / WHEN / THEN，可以转化为自动化测试。

- **它要实际检验其需求。** 一个只是用别的措辞重述需求、却什么都没检验的场景什么也测不到。让它成为一个有明确结果的具体情境。
- **覆盖重要的情形，而不仅是正常路径。** 合法登录很简单。空输入、过期令牌、第二次点击、出错的那个东西——bug 就藏在这些地方，场景的价值也在这里最高。
- **在标题中点名这个情形。** "Scenario: Rejects an expired token" 让审查者一眼看到覆盖了什么；"Scenario: Test 2" 则不能。

一个有用的习惯：在批准之前，问自己*哪个情形如果坏了我会最难受？*——并确保有一个场景点名了它。

## 选择正确的增量类型

一个变更用三种区段类型来描述它对规格的编辑。选对类型能让归档后的规格保持诚实：

- **`## ADDED Requirements`** —— 之前不存在的全新行为。
- **`## MODIFIED Requirements`** —— 已经存在并正在变化的行为。包含完整的新版本；一句关于改了什么的简短说明对审查者有帮助。
- **`## REMOVED Requirements`** —— 即将移除的行为，附上一行说明原因。

归档时，ADDED 会被追加到主规格，MODIFIED 会替换旧版本，REMOVED 会被删除。如果你把一个真实变更标为 ADDED，就会得到两条互相竞争的需求；如果你把新行为描述为 MODIFIED，却没有任何东西可以替换。拿不准时，打开当前规格看看那条需求是否已经在那里了。

## 给变更定好大小

最常见的撰写错误不是措辞糟糕的需求——而是一个变更试图同时做成三个变更。

**一个好变更只有一个你能用一句话讲清的意图。** "添加深色模式开关。""给登录端点做限流。""把会话从 cookie 迁移走。"如果描述这个变更需要很多"以及"，那就是该拆分的信号。

变更过大的迹象：

- 提案的范围读起来像一串互不相关的功能。
- 审查它要花一下午，所以没人会去做。
- 两个人没法在不冲突的情况下同时做它。
- 一半的任务本可以自己单独发布。

更小的变更更容易审查，更容易在一次专注的会话中完成，也更容易在六个月后当只剩归档时进行推理。你完全可以并行运行多个变更——参见[编辑与迭代](editing-changes.md)和[工作流](workflows.md)。

相反的情况也会发生：一行字的拼写修正不需要三条需求加一份设计文档。让仪式匹配利害关系。

## 如何引导 AI 走向好的草稿

因为 `/opsx:propose` 做的是第一版草稿，你拿回来的东西的质量取决于你给它的东西的质量。你不必亲手写需求——你只需把 AI 瞄准好：

- **陈述意图和边界。** *"添加一个深色模式开关，首次加载时跟随系统设置——不要动现有的主题 API。"* 范围之外的那一半和范围之内的那一半同等重要。
- **点名你在意的情形。** *"确保有一个为已经手动选择过主题的用户准备的场景。"* AI 覆盖的是你指给它看的东西。
- **然后编辑。** 它就是纯 Markdown。收紧一条含糊的 `SHALL`，删掉一个什么都没测的场景，补上它漏掉的情形——或者让 AI 来做：*"超时需求太含糊，把它固定为 30 分钟。"*

起草、打磨、重复。这样几轮下来就会产生一份你愿意信任的规格，而这正是全部意义所在。

## 一份快速清单

- [ ] 每条需求都是一个可观察的行为，带一个 `SHALL`/`MUST`。
- [ ] 没有任何实现细节被烤进需求里。
- [ ] 每条需求都至少有一个真正检验它的场景。
- [ ] 重要的边界和错误情形都有场景，而不仅仅是正常路径。
- [ ] 增量对当前规格正确地使用了 ADDED / MODIFIED / REMOVED。
- [ ] 整个变更只有一个你能用一句话陈述的意图。
