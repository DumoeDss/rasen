# 技能编写指南

本仓库中编写与编辑技能的写作规范 —— 涵盖专家技能（`src/core/templates/experts/<name>.ts`）和 OpenSpec 工作流技能。它是一份仓库文档，而非可安装的技能：它没有模板、不参与注册，也不会影响技能计数。

技能的存在，是为了从随机系统中驯服出确定性。**可预测性（Predictability）** —— 即 agent 每次运行走相同的*流程*，而非产出相同的输出 —— 是根本美德；下面每一条杠杆都在为它服务。

## 调用方式

技能有两种触达方式，选择哪一种，是在两种不同的成本之间做权衡：

- **模型触发（model-invoked）** 的技能保留 `description`，因此 agent 可以自主触发它，*并且*其他技能也能触达它（你仍然可以手动输入它的名字）。它会产生**上下文负担（context load）** —— 描述每轮都占据着窗口。操作方式：省略 `disable-model-invocation`，并编写面向模型的描述，包含丰富的触发措辞（"Use when the user wants…, mentions…"）。
- **用户触发（user-invoked）** 的技能把描述从 agent 的触达范围中剥离：只有手动输入其名字的人才能调用它，其他技能都不能。零上下文负担，但它要耗费**认知负担（cognitive load）** —— 人就是那个必须记住它存在的索引。操作方式：设置 `disable-model-invocation: true`；`description` 变成面向人的一行摘要，去掉触发短语列表。

只有当 agent 必须自行触达该技能，或另一个技能必须触达它时，才选择模型触发。如果它只会被手动触发，就把它设为用户触发，不付任何上下文负担。当用户触发的技能多到你记不住时，用一个**路由技能（router skill）**来化解堆积的认知负担：一个用户触发的技能，列出其他技能以及各自该在何时使用。

## 编写 description

模型触发的 `description` 同时承担两项任务 —— 说明技能是什么，并列出应触发它的**分支（branch）**。每个字都会增加上下文负担，因此 description 值得比正文更狠的精简：

- **把技能的引领词（leading word）前置** —— description 正是它发挥触发作用的地方。
- **每个分支一个触发短语。** 为同一分支换说法的同义词属于**重复（duplication）** —— "build features using TDD … asks for test-first development" 是把一个分支写了两遍。合并它们，只保留真正不同的分支。
- **删掉正文里已有的身份说明。** 让 description 只剩触发短语，再加上任何"当另一个技能需要……"的触达条款。

## 信息层级

技能由两类内容构成 —— **步骤（step）** 和 **参考（reference）** —— 二者自由混合。核心决策是用哪一种、各自位于**信息层级（information hierarchy）**的哪一档，这是一把梯子，按 agent 需要该材料的紧迫程度排序：

1. **技能内步骤（In-skill step）** —— `SKILL.md` 中的一条有序动作，是主层级。每个步骤以一条**完成判据（completion criterion）**收尾，即告诉 agent 工作已完成的条件。要让它*可检验*（agent 能否区分"完成"与"未完成"？），并在要紧处做到*穷尽*（"每一个被改动的模型都已核对"，而非"列出一个变更清单"）—— 模糊的判据会招致**过早完成（premature completion）**。
2. **技能内参考（In-skill reference）** —— `SKILL.md` 中的一条定义、规则或事实，按需查阅。它常常是合理的扁平同级集合（一次 review 的每条规则都放在同一档）—— 这是一种不错的安排，不是坏味道。
3. **外部参考（External reference）** —— 把参考内容从 `SKILL.md` 推出到一个独立文件（sidecar，如 `GLOSSARY.md` 或 `DEEPENING.md`），通过一条**上下文指针（context pointer）**触达，只有在该指针触发时才加载。

推下去太少，顶部就会臃肿；推下去太多，又会藏起 agent 真正需要的材料。这股张力就是整个决策。**渐进式披露（progressive disclosure）** —— 把参考内容从 `SKILL.md` 移到一个链接文件里 —— 是顶部保持可读的方法。分支化是最干净的披露测试：每个分支都需要的就内联，只有部分分支触达的就推到指针背后。一条上下文指针的*措辞*（而非它的目标）决定了 agent 何时、多可靠地触达该材料。

梯子决定的是某块内容*放多深*，而**同址（co-location）**决定的是它到位后*旁边放什么*：把一个概念的定义、规则和注意事项收在同一个标题下，而不是散落各处，这样读到一部分时，它的邻居也随之而来。

## 何时拆分

**粒度（granularity）**是你切分技能的细致程度，而每一刀都要花掉两种负担之一，所以只在切分物有所值时才拆：

- **按触发方式拆** —— 当你有一个独特的**引领词**应当独立触发它，或另一个技能必须触达它时，就拆出一个模型触发的技能。你要为新这条常驻的 description 付出上下文负担，所以这种独立触达必须值得。
- **按顺序拆** —— 当后面还没做的步骤（某一步的**后置步骤（post-completion steps）**）会诱惑 agent 草草赶完眼前这一步时，就把一串步骤拆开。让它们留在视野之外，能鼓励 agent 在当前任务上多做一些**前置功课（legwork）**。

## 引领词

**引领词（leading word）**是一个早已存在于模型预训练中的紧凑概念，agent 在运行技能时用它来思考（例如 *lesson*、*fog of war*、*tracer bullets*、*seam*、*deep module*、*ubiquitous language*）。作为 token 在全文中反复出现（尽管一个强引领词可能只需用一次），它积累出一个分布式的定义，并通过征用模型已有的先验，用最少的 token 锚定一整片行为区域。

它从两处服务于可预测性。在正文里它锚定*执行*：每当该词出现，agent 都触达同一种行为。在 description 里它锚定*触发*：当同一个词也活在你的 prompt、文档和代码中时，agent 会把这种共享语言和技能关联起来，从而更可靠地触发它。

自造一个词也行，只要你定义清楚，但生造的词征用不到任何先验 —— 你要用定义 token 去偿还一个预训练词免费提供的东西。优先去够一个已有的词。主动寻找把技能重构到引领词上的机会：在三个地方各自拼写出的三元组，或 description 里花一整句话去比划一个意思 —— 这都是亟待坍缩成单个 token 的段落。

- "fast, deterministic, low-overhead" → _tight_（一个 _tight_ 的循环）。
- "a loop you believe in" → _red_（这个循环在 bug 上变 _red_，否则不变）。

## 修剪

让每一个含义只守着一个**单一事实源（single source of truth）**：唯一权威的位置，这样改变行为只需改一处。

逐行检查**相关性（relevance）**：它是否仍然关乎这个技能在做什么？

然后逐句而非逐行地猎杀**无操作（no-op）**：对每一句单独跑一遍 no-op 测试，一句不通过，就整句删掉，而不是从里面抠字。下手要狠 —— 大多数不通过的散文应当被删，而非被重写。

## 失效模式诊所

用这些来诊断技能的问题。

- **过早完成（premature completion）** —— 在一步真正完成之前就收尾，注意力滑向了*"完成"*本身。防御手段，按顺序：先磨利完成判据（便宜、局部）；只有当它无可救药地模糊、*并且*你观察到 agent 在赶工时，才通过拆分顺序把后置步骤藏起来。
- **重复（duplication）** —— 同一个含义出现在不止一个地方。它要付出维护成本和 token，还把某个含义在梯子上的显眼程度抬高到超过它真实的位置。它恰恰是引领词的反面 —— 那是*有意地*靠重复一个 token 来提升注意力，而非重复含义。
- **沉积（sediment）** —— 因为"加感觉安全、删感觉有风险"而沉淀下来的陈旧层。任何缺乏修剪纪律的技能，最终都会落得这个下场。
- **蔓延（sprawl）** —— 一个技能单纯地太长了，哪怕每一行都是鲜活且唯一的。解药是那把梯子：把参考内容披露到指针背后，并按分支或顺序拆分，让每条路径只承载它需要的东西。
- **无操作（no-op）** —— 一行模型默认就会遵守的话，于是你白白付出负担却什么也没说。测试方法是：相比默认行为，它是否改变了行为？一个弱引领词（agent 已经偏 thorough 时还说 *be thorough*）就是一个 no-op；解法是换一个更强的词（*relentless*），而不是换技术。这是相对于模型的 —— no-op 之争要靠运行技能来裁定，而非靠辩论。
